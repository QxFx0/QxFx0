Ниже — integration-ready драфт под текущий pipeline.
Сразу 4 недостающих артефакта, из-за которых `CanonicalMoveFamily` ниже даю как **surrogate mapping**:

1. точный enum `CanonicalMoveFamily`;
2. текущий список `ParserKeywords` и их приоритеты;
3. текущий список `focus_stopwords` / `focus_markers`;
4. SQL schema для noun morphology.

Потому ниже:

* `CMClarify / CMGround / CMDistinguish / CMRepair` — это **routing hints**, совместимые с примерами из твоего запроса;
* модальные слова типа `могу / должен / can / should / must / may` почти везде трактуются как **focus_stopword** или **weak semantic cluster**, а не как сильный routing trigger;
* `не могу / cannot / forced` **не открывают `CMRepair` сами по себе**.

# Agency, Obligation & Rights Lexicon Research

## Legend

### Semantic role

* `PK` = parser_keyword
* `FM` = focus_marker
* `FS` = focus_stopword
* `SC` = semantic_cluster
* `MO` = morphology support

### Routing hint

* `AF` = agency found / volition / choice
* `AL` = agency lost / incapacity
* `OB` = obligation / duty
* `PM` = permission / allowed-forbidden
* `RT` = right / entitlement
* `RS` = responsibility / blame
* `BD` = boundary / limit
* `CR` = coercion / force
* `CN` = consent / assent
* `MX` = mixed conflict / contrast

### Risk

* `low` = safe standalone cue
* `med` = usable, but needs local context
* `high` = ambiguous; do not route on token alone

### Operational rule for `CMRepair`

```text
if lexicon_hit in {AL, CR} and distress_hit == 0
  then suppress CMRepair

if lexicon_hit in {AL, CR} and distress_hit > 0 and help_hit > 0
  then allow CMRepair candidate

if lexicon_hit in {OB, PM, RT, RS, BD, MX} and distress_hit == 0
  then suppress CMRepair
```

### Minimal distress/help gate

Без нового classifier, только лексический gate.

* `distress_hit` examples: `страшно`, `невыносимо`, `больно`, `паника`, `ужасно`, `не справляюсь`, `I panic`, `I am overwhelmed`
* `help_hit` examples: `помогите`, `help`, `please help`, `что делать срочно`

## 4. Focus Markers

### 4.1 Focus markers: modal/operator should not be focus, X should be focus

| phrase template                         | focus target                  |
| --------------------------------------- | ----------------------------- |
| `могу ли я X`                           | `X`                           |
| `должен ли я X`                         | `X`                           |
| `имею ли я право X`                     | `X`                           |
| `обязан ли я X`                         | `X`                           |
| `можно ли мне X`                        | `X`                           |
| `где граница между X и Y`               | pair `X/Y`                    |
| `это принуждение или выбор`             | pair `принуждение/выбор`      |
| `это согласие или давление`             | pair `согласие/давление`      |
| `can I X`                               | `X`                           |
| `should I X`                            | `X`                           |
| `do I have the right to X`              | `X`                           |
| `am I obligated to X`                   | `X`                           |
| `where is the boundary between X and Y` | pair `X/Y`                    |
| `is this coercion or choice`            | pair `coercion/choice`        |
| `is this consent`                       | event/object under evaluation |

### 4.2 Focus stopwords

Это токены, которые **почти никогда** не должны становиться focus в ordinary routing.

#### Hard focus stopwords

`могу`, `можно`, `нельзя`, `должен`, `обязан`, `should`, `must`, `can`, `may`, `allowed`

#### Weak focus stopwords

`право`, `right`, `обязанность`, `obligation`, `долг`, `duty`

Их можно делать focus только при meta-режиме:

* `что значит право`
* `what is duty`
* `объясни слово "обязан"`

### 4.3 Local focus rule

```text
if pattern matches:
  (могу ли я|должен ли я|обязан ли я|можно ли мне|имею ли я право|can I|should I|may I|am I obligated to|do I have the right to)
and an infinitive / verb-like token / concrete object follows
then
  modal/operator token := focus_stopword
  following action/object := proposition focus
```

## 6. What Not To Add

Это будет ложной сложностью для текущего pipeline:

1. **legal judgement engine**
   Нельзя из `имею право` делать вывод о реальном правовом статусе.

2. **full deontic logic**
   Не нужно моделировать `obligatory / permitted / forbidden` как полноценную логику с вложениями.

3. **moral verdict classifier**
   `должен` и `справедливо` не должны превращаться в moral-judgement subsystem.

4. **therapy/safety classifier**
   `я не могу` не должен автоматически запускать терапевтический маршрут без явных distress/help lexemes.

5. **policy compliance engine**
   `запрещено` в пользовательском тексте — это лексема, а не источник truth о policy.

6. **parser rewrite / scope engine**
   Не надо пытаться вычислять сложные вложенные модальности и разрешения.

7. **LLM-as-router / ML classifier**
   Для этой задачи достаточно lexical frames + focus shifting + suppression guards.

### Короткий operational вывод

Для QxFx0 здесь нельзя делать route по отдельным токенам `могу / должен / право / can / should / must`.
Нужна такая дисциплина:

1. **single modal/operator token** → `focus_stopword` или слабый `semantic_cluster`;
2. **multiword frame** → `ParserKeyword`;
3. **conflict clause** (`могу, но не должен`) → `CMDistinguish`;
4. **agency lost / coercion** → не `CMRepair` по умолчанию;
5. `CMRepair` открывается только при совместном наличии:

   * `AgencyLost` или `Coercion`,
   * `distressHit`,
   * `helpHit`.

## 4. Focus Markers

### 4.1 Marker rules

| phrase pattern                      | focus rule       | relation words that must not become focus | family bias          |
| ----------------------------------- | ---------------- | ----------------------------------------- | -------------------- |
| `что такое X`                       | focus = `X`      | `что`, `такое`                            | CMDefine             |
| `X следует из Y`                    | focus = `X`, `Y` | `следует`, `из`                           | CMGround             |
| `следует ли X из Y`                 | focus = `X`, `Y` | `следует`, `ли`, `из`                     | CMGround / CMClarify |
| `чем X отличается от Y`             | focus = `X`, `Y` | `чем`, `отличается`, `от`                 | CMDistinguish        |
| `X является необходимым условием Y` | focus = `X`, `Y` | `является`, `необходимым`, `условием`     | CMDistinguish        |
| `найди контрпример к X`             | focus = `X`      | `найди`, `контрпример`, `к`               | CMConfront           |
| `prove X from Y`                    | focus = `X`, `Y` | `prove`, `from`                           | CMGround             |
| `does X follow from Y`              | focus = `X`, `Y` | `does`, `follow`, `from`                  | CMGround             |
| `difference between X and Y`        | focus = `X`, `Y` | `difference`, `between`, `and`            | CMDistinguish        |

### 4.2 Focus stopwords

Эти токены помогают routing, но **не должны становиться focus**:

`если`, `то`, `следовательно`, `значит`, `следует`, `влечёт`, `логически`, `if`, `then`, `therefore`, `hence`, `thus`.

### 4.3 Minimal extraction rules

1. Если сработал `focus_marker`, focus = ближайшие content spans справа/по обе стороны, а не connective.
2. Relation words `следует`, `влечёт`, `follows`, `implies`, `entails` записываются как relation tag, но получают `focus_penalty`.
3. Standalone `if/then/если/то` не поднимают family выше `CMClarify` без соседних logic nouns/phrases.
4. Пары `X/Y` при `difference/equivalence/necessary condition` должны извлекаться как dual focus.

### 7.2 ParserKeywords

Добавить exact/multi-word first:

1. **High-priority exact phrases**

   * `если и только если`
   * `тогда и только тогда`
   * `следует из`
   * `не следует из`
   * `влечёт за собой`
   * `не влечёт`
   * `вытекает из`
   * `necessary condition`
   * `sufficient condition`
   * `necessary and sufficient`
   * `if and only if`
   * `follows from`
   * `does not follow`
   * `proof by contradiction`

2. **Do not up-rank standalone connectives**

   * `если`, `то`, `if`, `then`, `therefore`, `thus`, `hence`
   * mark as `focus_stopword`, not strong parser trigger

### 7.3 Semantic clusters

Предлагаемые lightweight clusters:

* `InferenceSupport`

  * посылка, вывод, premise, conclusion, follows from, entails, derive
  * bias: `CMGround`

* `ConditionRelation`

  * necessary condition, sufficient condition, if and only if, равносильность
  * bias: `CMDistinguish`

* `RefutationConflict`

  * контрпример, противоречие, несовместимость, does not follow, contradiction
  * bias: `CMConfront`

* `FormalObject`

  * аксиома, теорема, лемма, rule of inference, quantifier, predicate
  * bias: `CMDefine`

* `QuantifiedForm`

  * все, каждый, существует, for all, there exists, at least one
  * bias: `CMClarify`

### 7.4 Focus logic

Rules:

1. If `focus_marker` matched, extract nearest noun/proposition spans as focus.
2. Apply strong negative weight to:

   * `если`, `то`, `if`, `then`, `therefore`, `thus`, `hence`, `следовательно`, `логически`
3. Relation words:

   * `следует`, `влечёт`, `follows`, `implies`, `entails`
   * store as relation tags, but **not focus**
4. For paired patterns:

   * `difference between X and Y`
   * `чем X отличается от Y`
   * `X является необходимым условием Y`
     focus must be pair `X/Y`.

### 7.5 `runSemanticLogic`

Cheap scoring only:

* `CMGround +=` derivation/support terms and exact entailment phrases
* `CMDistinguish +=` equivalence/condition contrast terms
* `CMConfront +=` non-entailment/refutation/contradiction terms
* `CMDefine +=` formal object terms and `what is` shells
* `CMClarify +=` quantified forms and generic `верно ли/does it follow` shells when relation underspecified

Hard constraint:

```text
logic_lexicon_hit alone
=> never CMRepair
```

### 7.6 Golden tests

Use the 40 JSONL cases above as regression corpus.
Add 3 invariant assertions in runner:

1. `focus_stopword` must not become primary focus.
2. exact non-entailment phrases must prefer `CMConfront` over `CMGround`.
3. paired condition terms must prefer `CMDistinguish` over `CMDefine`, unless shell is purely definitional (`что такое necessary condition`).

### 7.7 Migration-safe rollout

1. Seed SQL tables.
2. Add phrase-trigger pass before generic token scoring.
3. Add focus-stopword penalties.
4. Add cluster scores with low initial weights.
5. Run golden corpus.
6. Only then increase phrase-trigger priority above generic relation words.

Главный инвариант:

```text
multi-word logical phrase > single relation word
content noun pair > connective
logic routing > CMRepair, unless there is an independent distress gate
```

## 4. Focus Markers

Поскольку текущий `extractFocusEntity` склонен брать первый noun-like token, а `buildPrepareEffectPlan` затем пробует `ipfFocus*`, atom focus и только потом `ssLastTopic`, правила фокуса должны быть **pattern-first**: выделять объект справа, пару `X|Y`, либо хост после предлога, а для zero-object acts (`продолжи`) — отдавать фокус предыдущей теме. 

| pattern                               | focus rule                                                 | expected focus            | must_not_focus                         | note                                |                                    |                               |
| ------------------------------------- | ---------------------------------------------------------- | ------------------------- | -------------------------------------- | ----------------------------------- | ---------------------------------- | ----------------------------- |
| `что такое X`                         | брать span справа от trigger                               | `X`                       | `что`, `такое`                         | strongest RU definitional rule      |                                    |                               |
| `объясни X`                           | брать правый object span                                   | `X`                       | `объясни`                              | simple imperative object rule       |                                    |                               |
| `уточни X`                            | брать правый object span                                   | `X`                       | `уточни`                               | wording/topic clarification         |                                    |                               |
| `что имеется в виду под X`            | брать span после `под`                                     | `X`                       | `что`, `имеется`, `виду`, `под`        | preposition host rule               |                                    |                               |
| `о чём речь в X`                      | брать span после `в` или quoted noun                       | `X`                       | `о`, `чём`, `речь`, `в`                | anchor-to-topic rule                |                                    |                               |

## 6. What Not To Add

Не добавлять distinctions, которые будут ложной сложностью для текущего pipeline:

* отдельный intent family для каждого глагола (`explain` vs `clarify` vs `elaborate`);
* ML classifier или LLM-as-router;
* dependency parser ради `X/Y` pair extraction;
* theorem prover для `prove/refute/counterexample`;
* новый discourse planner ради `continue / summarize / next step`;
* тонкие прагматические классы вроде `polite_ask`, `firm_ask`, `soft_refute`, если они не меняют family;
* новый AST для imperative/wh-question layers;
* scoring engine, который “понимает” доказательство как формальную валидность;
* новые families для `example`, `rephrase`, `stepwise`, если они уже ложатся в `CMGround / CMClarify / CMDeepen`;
* broad rewrite `parseProposition` или `MeaningAtoms`.

Главный принцип: **углублять keyword tables и focus markers, а не таксономию архитектуры**.

## 7. Minimal Integration Plan

1. **Добавить strong multiword `ParserKeywords` раньше одиночных слов.**
   Это особенно важно, потому что `detectPropositionType` сейчас идёт по first-match keyword order. Сначала должны проверяться паттерны вроде `что такое`, `чем X отличается от Y`, `на каком основании`, `разложи по шагам`, `find the error in`, а уже потом — более общие verbs. 

2. **Разделить словарь на три слоя.**

   * `parser_keyword`: устойчивые imperative / interrogative phrases.
   * `focus_marker`: pair markers, preposition-host markers, shell-noun suppressors.
   * `morphology`: noun seeds с case-aware forms (`определение`, `объяснение`, `уточнение`, `различие`, `сравнение`, `обоснование`, `основание`, `ошибка`, `пример`, `шаг`, `резюме`).

3. **Добавить act-level stopwords и shell-noun suppression.**
   Для focus extraction и `extractObject` понизить вес: `что`, `такое`, `чем`, `от`, `между`, `и`, `ли`, `в`, `на`, `по`, `шагам`, `пример`, `ошибку`, `разница`, `сходство`, `следующий`, `term`, `definition`, `error`.
   Правило: если есть pattern-marker, topic должен браться **из объекта**, а не из shell noun.

4. **Ввести pair extraction и prepositional host rules без parser rewrite.**

   * `X отличается от Y` -> focus `X|Y`
   * `difference between X and Y` -> focus `X|Y`
   * `ошибка в X` -> focus `X`
   * `контрпример к X` -> focus `X`
   * `разложи X по шагам` -> focus `X`
   * `продолжи / what next` -> focus = `ssLastTopic`

5. **Добавить contrast preference rule.**
   Для `не X, а Y` и `not X but Y`:

   * понизить вес `X`;
   * повышать act/focus для `Y`;
   * не позволять первому keyword-попаданию захватывать family.

6. **Прогнать инвариантный corpus из раздела 5 как gate.**
   Проверять:

   * `expected_family_any_of`,
   * `must_not_family`,
   * `must_not_focus`,
   * `expected_force_any_of`,
   * `requires_replay_envelope = true`.

7. **Optional SQL seed only for nouns.**
   Не как новый classifier, а как маленький seed-table для case-aware lookup:

   * columns: `lemma`, `surface_form`, `lang`, `family_hint`, `semantic_role`, `case_tag`, `weight`, `enabled`
   * seed only noun rows: `определение`, `объяснение`, `уточнение`, `различие`, `сравнение`, `доказательство`, `обоснование`, `основание`, `опровержение`, `контрпример`, `ошибка`, `проверка`, `пример`, `шаг`, `резюме`
   * использовать только как вспомогательный lookup для `ParserKeywords`/focus, не как источник “истинного intent”.

Если нужно, следующим сообщением я превращу это в компактный **TSV для словаря** и отдельный **готовый JSONL corpus** без пояснительного текста.
# Emotion, Dialogue Rupture & Repair Lexicon Research

Ниже — lexical-only пакет для текущего QxFx0 pipeline `raw input -> parseProposition -> collectAtoms -> runSemanticLogic -> runFamilyCascade -> render/finalize`. Это сознательно не новый intent engine: в пакете видно, что `parseProposition` сейчас делает first-match keyword typing, `ipfIsNegated` уже есть, но используется непоследовательно, `MeaningAtoms` ловит distress/contact через lexical triggers, а `runSemanticLogic` уже маппит `Exhaustion -> CMRepair`, `NeedContact -> CMContact`, `NeedMeaning -> CMReflect`, `Verification -> CMClarify`, `Contradiction -> CMConfront`. Значит, правильный ход здесь — расширять словарь, stopwords/markers и локальные negation/contrast rules, а не строить новую архитектуру.

## 4. Focus Markers

### Phrases where focus should shift to the topic after the marker

* `вернись к X`
* `вернись к вопросу о X`
* `ты не ответил на X`
* `я спрашивал про X`
* `это не X, а Y` → **focus = Y**
* `помоги с X`
* `объясни X, я не понимаю`
* `почини ответ про X`
* `восстанови связь по X`
* `return to X`
* `you did not answer X`
* `I asked about X`
* `help me with X`
* `explain X, I do not understand`
* `repair the answer about X`

### Focus markers

* `вернись к`
* `я спрашивал про`
* `ты не ответил на`
* `помоги с`
* `объясни`
* `разбери`
* `не X, а Y`
* `return to`
* `I asked about`
* `you did not answer`
* `help me with`
* `explain`
* `not X but Y`

### Focus stopwords

Обязательный минимальный набор:

* `устал`
* `одиноко`
* `помоги`
* `пожалуйста`
* `слышишь`
* `не`
* `просто`
* `this`
* `help`
* `tired`
* `lonely`

Расширенный безопасный набор для этой фазы:

* RU: `я`, `мне`, `ты`, `это`, `здесь`, `просто`, `вообще`, `сейчас`, `пожалуйста`, `слышишь`, `устал`, `одиноко`, `поддержи`, `помоги`, `вернись`
* EN: `I`, `me`, `you`, `this`, `just`, `please`, `hear`, `help`, `tired`, `lonely`, `repair`, `answer`

### Relation/content rules

* `не X, а Y` → down-rank `X`, up-rank `Y`
* `ты не ответил на X` → prefer `X`, not `ответил`
* `помоги с X` → prefer `X`, not `помоги`
* `объясни X, я не понимаю` → prefer `X`, not `объясни` or `понимаю`
* `you are not listening about X` → if `X` present, prefer `X`, else treat as rupture
* `I asked about X, not Y` → prefer `X` and `Y`, not `asked`

## 6. What Not To Add

Не добавлять в этой фазе:

* diagnosis labels (`depression`, `anxiety disorder`, `attachment style`, и т.п.)
* clinical severity scoring
* self-harm policy classifier
* therapy classifier
* personality typing
* deep emotion taxonomy, выходящую за routing needs
* новый intent engine
* parser rewrite
* LLM-as-router
* separate mental-health ontology
* surface-level “emotion score” без связи с route invariants

Достаточно отличать:

* distress vs negated-distress
* contact need vs rupture
* disagreement vs support request
* affect word as content vs affect word as route signal
* logical request with emotional vocabulary vs actual CMRepair case

## 7. Minimal Integration Plan

Пакет должен встраиваться только в существующие lexical seams. Это согласуется с кодом pack: `detectPropositionType` — first-match keyword selection, `lexicalAtoms` already injects `Exhaustion`, `NeedContact`, `NeedMeaning`, `AgencyLost`, а `runSemanticLogic` уже маппит эти атомы в `CMRepair`, `CMContact`, `CMReflect`, `CMClarify`, `CMConfront`. Поэтому ниже — только локальные изменения словаря и suppression rules, без новой runtime boundary.  

### ParserKeywords

Добавить словарь в существующие keyword lists:

* **repair keywords**
  `устал`, `выгорел`, `нет сил`, `не могу больше`, `опустились руки`, `exhausted`, `burned out`, `no energy`, `can’t go on`
* **contact keywords**
  `одиноко`, `не с кем поговорить`, `побудь рядом`, `слышишь меня`, `мне нужен контакт`, `lonely`, `hear me`, `stay with me`, `I need connection`
* **rupture keywords**
  `ты меня не слышишь`, `ты уходишь от темы`, `это не то, что я спросил`, `вернись к вопросу`, `почини ответ`, `you are not listening`, `repair the answer`, `return to the question`
* **confront keywords**
  `не согласен`, `возражаю`, `это неверно`, `ты ошибаешься`, `I disagree`, `this is wrong`, `I object`
* **anti-repair / clarify markers**
  `не устал`, `не выгорел`, `просто уточняю`, `хочу уточнить`, `not tired`, `this is a logic question`

### MeaningAtoms lexical triggers

Расширить `lexicalAtoms` и phrase-level suppression:

* добавить positive distress/contact/rupture lexemes
* добавить **negated distress override set**:

  * `не устал`
  * `не выгорел`
  * `не измучен`
  * `not tired`
  * `not exhausted`
* добавить **contrast override**:

  * `не X, а Y` → не создавать тот же atom для отвергнутого `X`
* добавить **repair-topic markers**:

  * `вернись к X`
  * `я спрашивал про X`
  * `ты не ответил на X`
* support verbs (`помоги`, `help me`) не должны автоматически форсить `CMRepair`, если у них есть явный логический объект (`посылка`, `аргумент`, `premise`, `conclusion`)

### Focus stopwords / markers

Расширить `Proposition.hs`:

* **focus stopwords**

  * `устал`, `одиноко`, `помоги`, `пожалуйста`, `слышишь`, `не`, `просто`, `this`, `help`, `tired`, `lonely`
* **focus markers**

  * `вернись к`
  * `я спрашивал про`
  * `ты не ответил на`
  * `помоги с`
  * `объясни X`
  * `return to`
  * `I asked about`
  * `you did not answer`
  * `help me with`

### Semantic clusters

Минимально расширить кластеры, не добавляя нового intent engine:

* `DistressExhaustion`
* `ContactNeed`
* `DialogueRupture`
* `RepairRequest`
* `DistrustDoubt`
* `DisagreementError`
* `AffectSupport`
* `AntiRepairClarify`

Они должны только bias existing families:

* `CMRepair`
* `CMContact`
* `CMReflect`
* `CMClarify`
* `CMConfront`
* `CMDistinguish`

### Invariant corpus

Положить corpus в existing semantic golden/invariant harness:

* 40 cases above
* проверять:

  * `must_not_family`
  * `expected_family_any_of`
  * `must_not_focus`
  * `must_not_guard_reason_contains`
  * `requires_replay_envelope`

Не проверять:

* полный rendered text
* психологические интерпретации
* “правильность терапии”
* deep diagnosis output

Главный практический эффект этого пакета:

1. удержать реальные distress/contact/rupture сигналы в `CMRepair / CMContact / CMReflect`;
2. не отправлять negated distress и логические запросы в `CMRepair`;
3. различать disagreement/error claims от support requests;
4. сделать `не X, а Y` и repair-markers полезными для focus extraction без расширения архитектуры.
