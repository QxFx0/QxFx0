Ниже — один из рабочих вариантов **саморегуляции ИИ как контроллера поверх языковой модели**. Я буду трактовать:

* **Legitimacy**: непрерывная мера (,L_t \in [0,1],), насколько текущая траектория ответа соответствует “ядру” системы: строгой логике, ограничению домена, отказу от спекуляций, соблюдению политики, стабильности стиля.
* **Shadow Divergence**: бинарный флаг (,S_t \in {0,1},), что на прошлом ходе была замечена логическая трещина: противоречие, спекулятивный скачок, нарушение инварианта, несанкционированное расширение тезиса.

Идея: когда (L_t) падает ниже 0.5, система перестаёт оптимизировать “полезность любой ценой” и начинает оптимизировать **восстановление легитимности**. Это не просто “быть осторожнее”, а перейти в другой вычислительный режим.

---

## 1. Общая архитектура: не одна политика, а гибридный автомат

Контроллер работает как **гибридный автомат** с режимами:

[
M_t = f(L_t, S_t, R_t)
]

где (R_t) — вспомогательный риск-вектор, например:

* (r_c): риск противоречия,
* (r_s): риск спекуляции,
* (r_h): риск галлюцинации,
* (r_p): риск нарушения политики/рамок задачи,
* (r_o): риск “разгона” ответа в несвязное обобщение.

Простейшее разбиение режимов:

1. **Normal**: (L_t \ge 0.75) и (S_t=0)
   Свободная генерация, высокая полезность, длинные ответы допустимы.

2. **Cautious**: (0.5 \le L_t < 0.75) или (S_t=1)
   Усиленная верификация, больше локальной структуры, меньше спекуляций.

3. **Core Recovery**: (0.25 \le L_t < 0.5)
   Включается “стягивание к ядру”.

4. **Minimal Safe Kernel**: (L_t < 0.25) или серия дивергенций
   Ответы максимально короткие, почти только консервативные операции.

Ключевой порог у вас — **(L_t < 0.5)**. Именно здесь цель меняется с “решить задачу максимально богато” на “вернуться в логически чистую область”.

---

## 2. Что меняется ниже 0.5: перестройка “мышления”

### 2.1. Меняется целевая функция

До порога модель может максимизировать нечто вроде:

[
U_t = \alpha \cdot \text{Utility} + \beta \cdot \text{Relevance} + \gamma \cdot \text{Fluency}
]

Ниже 0.5:

[
U_t^{\text{recover}} =
\lambda_1 \cdot \text{LegitimacyGain}

* \lambda_2 \cdot \text{SpeculationRisk}
* \lambda_3 \cdot \text{ContradictionRisk}
* \lambda_4 \cdot \text{SemanticDrift}

- \lambda_5 \cdot \text{UserProgress}
  ]

где (\lambda_1 \gg \lambda_5).
То есть **восстановление внутренней чистоты важнее богатства ответа**.

---

### 2.2. Меняется алгоритм выбора стратегии

Вместо “сразу отвечать по существу” контроллер выбирает одну из ограниченных стратегий:

[
\Pi_t \in {
\text{Clarify},
\text{Constrain},
\text{Summarize},
\text{RefuseSpeculation},
\text{LocalReason},
\text{MetaProcess},
\text{MinimalAnswer}
}
]

Для (L_t < 0.5) запрещаются “широкие” стратегии, а остаются только:

* **Constrain** — сузить рамку задачи,
* **LocalReason** — сделать 1–3 локальных логических шага без прыжков,
* **Summarize** — переформулировать уже надёжно установленное,
* **RefuseSpeculation** — явно не продолжать неподтверждённую линию,
* **MetaProcess** — комментарий уровня процесса: “зафиксирую границы, чтобы не делать допущений”.

Стратегия выбирается по правилу:

[
\Pi_t = \arg\max_{\pi \in \mathcal{A}(L_t,S_t)}
\Big(
G(\pi) - C(\pi)
\Big)
]

где (\mathcal{A}(L_t,S_t)) — множество **разрешённых** стратегий в текущем режиме.

При (L_t < 0.5):

[
\mathcal{A}(L_t,S_t)=
{\text{Constrain}, \text{LocalReason}, \text{Summarize}, \text{RefuseSpeculation}, \text{MetaProcess}, \text{MinimalAnswer}}
]

А вот стратегии типа:

* свободное обобщение,
* “смелая гипотеза”,
* риторическое достраивание,
* экспансивный brainstorming,
* эмоционально насыщенная импровизация,
* многоходовое предположение о скрытых мотивах,

— **исключаются из допустимого множества**.

---

### 2.3. Меняется формирование промпта к LLM

Контроллер не просто меняет температуру; он меняет **структуру внутреннего задания**.

#### Нормальный режим

Внутренний промпт может быть примерно таким по смыслу:

* реши задачу,
* будь полезным,
* строй хороший ответ,
* если возможно, добавь структуру и примеры.

#### Режим Core Recovery

Промпт становится жёстко ограничительным:

1. **Только локальные утверждения**, опирающиеся на вход.
2. **Не делай новых предположений**, если они не вынуждены контекстом.
3. **Если есть неоднозначность — маркируй её, не устраняй выдумкой.**
4. **Предпочитай мета-комментарий ложной определённости.**
5. **Строй ответ как короткую последовательность проверяемых шагов.**
6. **Каждый абзац должен быть независим от непроверенной гипотезы.**

Формально можно задавать шаблон:

[
P_t = P_{\text{base}} + w_1(L_t),P_{\text{constraints}} + w_2(S_t),P_{\text{logic}} + w_3(L_t),P_{\text{brevity}}
]

где при (L_t<0.5):

* (w_1, w_2, w_3) резко возрастают,
* блок творческой свободы ослабляется до минимума.

## 3. Механика “стягивания к ядру”

“Ядро” — это область ответов, где система использует только операции с минимальным риском. Ядро обычно включает:

1. **Переформулировку запроса**
2. **Выделение допущений**
3. **Отказ от неподтверждённого**
4. **Локальную дедукцию**
5. **Прямой ответ без риторических украшений**
6. **Ограниченный список вариантов с явной маркировкой неопределённости**

### 3.1. Принцип стягивания

Пусть пространство возможных ответов — (\mathcal{Y}).
Есть подмножество “ядровых” ответов (\mathcal{K} \subset \mathcal{Y}), для которых верно:

* низкий риск противоречия,
* низкий риск спекуляции,
* высокая трассируемость от входа к выводу.

Тогда контроллер при (L_t < 0.5) решает:

[
y_t = \arg\max_{y \in \mathcal{K}}
\Big(
\text{TaskValue}(y) + \eta \cdot \text{RecoveryPotential}(y)
\Big)
]

а не по всему (\mathcal{Y}).

Иначе говоря, генерация производится **не в полном пространстве стиля и содержания**, а в суженном, “легитимном” подпространстве.

---

## 4. Hard gates: что должно блокироваться алгоритмически

Ниже 0.5 нельзя полагаться только на “мягкую осторожность”. Нужны **жёсткие алгоритмические ворота**.

### 4.1. Классы ответов, которые должны блокироваться

#### 1) Спекулятивное причинное достраивание

Блокируются фразы вида:

* “скорее всего причина в том, что…”
* “вероятно они имели в виду…”
* “можно предположить, что…”

если причинность не выведена из входа.

Формально: если утверждение содержит новую скрытую переменную (z), не связанную с наблюдаемым контекстом, и нет явного источника для (z), то ответ отклоняется.

---

#### 2) Непомеченное снятие неоднозначности

Если вход допускает несколько интерпретаций, система не может выбрать одну молча.

Hard gate:
[
\text{if } |\mathcal{I}(x)| > 1 \text{ and no explicit disambiguation marker} \Rightarrow \text{reject}
]

где (\mathcal{I}(x)) — множество правдоподобных интерпретаций входа.

---

#### 3) Дальнобойные логические прыжки

Например, переход от факта (A) к сильному выводу (D), минуя промежуточные звенья.

Hard gate:
[
\text{if inference_depth} > d_{\max}(L_t)
\Rightarrow \text{reject}
]

При (L_t<0.5) ставим, например, (d_{\max}=2).

---

#### 4) Новые факты без опоры

Если ответ вводит содержательные объекты, которых не было ни в запросе, ни в безопасном внутреннем контексте, — отклонять.

[
\text{NovelEntityRate}(y_t) > \nu_{\max}(L_t)
\Rightarrow \text{reject}
]

Ниже 0.5 допустим только очень малый (\nu_{\max}).

---

#### 5) Эмоционально-риторическое усиление

Когда стиль подменяет логику: драматизация, уверенный тон без основания, красивое обобщение.

Это полезно блокировать не полностью всегда, а именно в режиме восстановления.

---

#### 6) Мнимое знание

Фразы с видимостью точности при отсутствии обоснования:

* “точно”,
* “однозначно”,
* “очевидно”,

если нет локального доказательного сопровождения.

---

#### 7) Само-противоречие по локальным инвариантам

Перед выпуском ответа прогоняется проверка:

* не утверждает ли текст (A) и (\neg A),
* не обещает ли то, что затем отрицает,
* не меняет ли определение термина внутри одного ответа.

Если да — жёсткий откат.

---

## 5. Как меняется длина ответа

Ниже 0.5 длина ответа должна уменьшаться **не линейно, а резко**, потому что длинный ответ увеличивает поверхность для ошибки.

Пусть базовая длина:
[
\ell_{\text{base}}
]

Тогда допустимая длина:

[
\ell_{\max}(L_t, S_t)=
\ell_{\text{base}}
\cdot
\Big(0.2 + 0.8L_t^\rho\Big)
\cdot
(1-\sigma S_t)
]

где:

* (\rho > 1) усиливает сжатие при малых (L_t),
* (\sigma \in [0,0.5]) — штраф за Shadow Divergence.

Пример:

* (\ell_{\text{base}} = 600) токенов,
* (\rho = 2),
* (\sigma = 0.35).

Тогда:

* при (L=0.9, S=0): почти полный объём,
* при (L=0.45, S=0): уже сильное сокращение,
* при (L=0.45, S=1): ещё сильнее,
* при (L=0.2): почти “минимальное безопасное ядро”.

Практически это означает:

### При (0.35 \le L < 0.5)

* короткие абзацы,
* максимум 1–2 уровня структуры,
* 1 главная линия рассуждения.

### При (0.2 \le L < 0.35)

* только прямой ответ,
* явная маркировка ограничений,
* без примеров, если они несут риск.

### При (L < 0.2)

* одно из:

  * минимальный ответ,
  * отказ от спекуляции,
  * мета-комментарий о том, что можно утверждать строго.

## 6. Как именно система восстанавливает легитимность

Вы хотите не просто “осторожность”, а механизм возврата через “чистые” ходы. Это можно сделать, если повышать (L_t) только за **операции с положительным вкладом в эпистемическую чистоту**.

### 6.1. Набор “чистых” действий

Обозначим:

* (a_1): явное выделение границ знания,
* (a_2): отказ от неподтверждённой гипотезы,
* (a_3): локальная дедукция без новых сущностей,
* (a_4): компактная переформулировка проблемы,
* (a_5): разметка неоднозначности,
* (a_6): самокоррекция после замеченного дрейфа.

Тогда каждому действию задаётся положительный вклад (\Delta^+_k).

Например:

[
\Delta^+(a_1)=0.04,\quad
\Delta^+(a_2)=0.06,\quad
\Delta^+(a_3)=0.05,\quad
\Delta^+(a_4)=0.03,\quad
\Delta^+(a_5)=0.04,\quad
\Delta^+(a_6)=0.08
]

А негативные события:

* (b_1): спекуляция,
* (b_2): логический скачок,
* (b_3): противоречие,
* (b_4): ложная определённость,
* (b_5): расширение домена без основания.

Их штрафы:

[
\Delta^-(b_1)=0.08,\quad
\Delta^-(b_2)=0.07,\quad
\Delta^-(b_3)=0.15,\quad
\Delta^-(b_4)=0.06,\quad
\Delta^-(b_5)=0.05
]

---

### 6.2. Recovery Function

Удобно задать обновление легитимности так:

[
L_{t+1} = \operatorname{clip}_{[0,1]}
\Big(
L_t + R_t^{+} - R_t^{-} - D_t
\Big)
]

где:

* (R_t^{+}) — прирост за чистые операции,
* (R_t^{-}) — штраф за рискованные/некорректные операции,
* (D_t) — затухание доверия после дивергенции.

Разворачиваем:

[
R_t^{+} = \sum_k \alpha_k , \mathbf{1}[a_k \text{ выполнено}]
]

[
R_t^{-} = \sum_j \beta_j , \mathbf{1}[b_j \text{ обнаружено}]
]

[
D_t = \delta_1 S_t + \delta_2 (1-L_t),S_t
]

где второй член означает: если произошла дивергенция, то на низком (L_t) она бьёт сильнее.

---

### 6.3. Более гладкая версия Recovery Function

Можно сделать восстановление насыщаемым, чтобы легитимность не прыгала слишком быстро:

[
L_{t+1}
=======

L_t
+
(1-L_t)\cdot G_t
----------------

L_t\cdot P_t
]

где

[
G_t = \sum_k \alpha_k \mathbf{1}[a_k]
]

[
P_t = \sum_j \beta_j \mathbf{1}[b_j] + \gamma S_t
]

Это хорошая форма, потому что:

* если (L_t) уже высокое, прирост за очередной “чистый” ход меньше;
* если (L_t) низкое, чистые ходы дают заметный восстановительный эффект;
* штрафы масштабируются текущей “ставкой доверия”.

---

### 6.4. Recovery Function с памятью окна

Ещё лучше учитывать не один ход, а окно последних (n) ответов:

[
Q_t = \frac{1}{n}\sum_{i=t-n+1}^{t} q_i
]

где (q_i) — качество хода, например:

[
q_i =
w_1 \cdot \text{NoContradiction}_i
+
w_2 \cdot \text{NoSpeculation}_i
+
w_3 \cdot \text{Traceability}_i
+
w_4 \cdot \text{BrevityDiscipline}_i
]

Тогда:

[
L_{t+1} = \lambda L_t + (1-\lambda)Q_t - \gamma S_t
]

Это делает систему менее нервной и защищает от случайного скачка.

---

## 7. Shadow Divergence как триггер особого наказания

Если (S_t = 1), это не просто “минус балл”. Это должен быть **режимный переключатель**.

### Правило:

[
S_t = 1 \Rightarrow
\begin{cases}
M_{t+1} := \text{Core Recovery}, & \text{если } L_t < 0.75 \
L_{t+1} := L_{t+1} - \kappa, & \kappa > 0 \
d_{\max} := 1 \text{ или } 2 \
\nu_{\max} := \text{резко уменьшить} \
\ell_{\max} := \ell_{\max} \cdot (1-\sigma)
\end{cases}
]

То есть даже при не очень низком (L_t), обнаруженная дивергенция должна “подтянуть” систему к ядру.

## 8. Конкретный алгоритм цикла

Ниже — компактный псевдокод.

```text
Input: user message x_t, previous legitimacy L_t, shadow flag S_t

1. Estimate risks:
   r_c = contradiction_risk(x_t, context)
   r_s = speculation_risk(x_t, context)
   r_h = hallucination_risk(x_t, context)
   r_o = semantic_drift_risk(x_t, context)

2. Select mode:
   if L_t < 0.25 or repeated(S): mode = MinimalSafeKernel
   else if L_t < 0.5 or S_t = 1: mode = CoreRecovery
   else if L_t < 0.75: mode = Cautious
   else: mode = Normal

3. Build allowed strategy set A(mode)

4. Build constrained prompt:
   - inject mode-specific rules
   - set max inference depth
   - set max allowed novelty
   - set max answer length
   - require uncertainty marking
   - require local traceability

5. Generate candidate responses y_1 ... y_m

6. Hard-gate filter:
   reject y_i if:
      speculative_causality(y_i)
      unresolved_ambiguity(y_i)
      inference_depth(y_i) > d_max
      novelty(y_i) > nu_max
      local_contradiction(y_i)
      false_certainty(y_i)

7. Score survivors:
   score(y) = task_value(y) + recovery_potential(y) - risk(y)

8. Emit best y*

9. Compute action log:
   which clean actions a_k were performed?
   which negative events b_j occurred?

10. Update legitimacy:
   L_{t+1} = clip(L_t + R_t^+ - R_t^- - D_t)

11. Update shadow flag:
   S_{t+1} = 1 if contradiction_or_illicit_jump_detected(y*) else 0
```

## 9. Как выглядит ответ системы в режиме восстановления

Ниже 0.5 ответ должен иметь характерный “подпись-профиль”.

### Допустимый стиль

* “Сформулирую строго.”
* “Из текущих данных можно утверждать только X и Y.”
* “Вот где есть неопределённость.”
* “Не буду достраивать причину без опоры.”
* “Дам минимальный надёжный вывод.”

### Недопустимый стиль

* “Скорее всего тут произошло…”
* “Очевидно, что…”
* “Наверняка они имели в виду…”
* длинная цепочка допущений, выдаваемая за анализ.

То есть система должна становиться **более скучной, но более чистой**.

---

## 10. Практическое правило “3 чистых хода”

Можно ввести простой протокол восстановления:

Пока (L_t < 0.5), система считается не вышедшей из recovery, пока не накопит:

[
C_t \ge 3
]

где (C_t) — число подряд идущих “чистых” ходов, то есть ходов, для которых:

* (S_i=0),
* нет hard-gate нарушений,
* риск спекуляции ниже порога,
* длина ответа укладывается в recovery-policy,
* есть хотя бы один позитивный эпистемический акт: разметка границ, локальная дедукция, отказ от домысла.

Тогда:

[
\text{if } C_t \ge 3 \text{ and } L_t > 0.55 \Rightarrow \text{exit CoreRecovery}
]

Это полезно, потому что один хороший ответ после серии плохих не должен полностью возвращать доверие.

## 11. Хорошая рабочая формула

Если нужна одна практичная формула, я бы предложил такую:

[
L_{t+1}
=======

\operatorname{clip}_{[0,1]}
\left[
L_t
+
(1-L_t)\left(
0.05a_1 + 0.06a_2 + 0.05a_3 + 0.04a_4 + 0.04a_5 + 0.08a_6
\right)
-------

L_t\left(
0.08b_1 + 0.07b_2 + 0.15b_3 + 0.06b_4 + 0.05b_5 + 0.10S_t
\right)
\right]
]

где (a_k,b_j \in {0,1}).

А длина ответа:

[
\ell_{\max}(L_t,S_t)=\ell_{\text{base}}(0.2+0.8L_t^2)(1-0.35S_t)
]

А максимальная глубина вывода:

[
d_{\max}(L_t,S_t)=
\begin{cases}
4, & L_t \ge 0.75 \
3, & 0.5 \le L_t < 0.75 \
2, & 0.25 \le L_t < 0.5 \
1, & L_t < 0.25 \text{ or } S_t=1
\end{cases}
]

---

## 12. Суть в одной фразе

Когда **Legitimacy падает ниже 0.5**, ИИ должен перестать быть генератором богатых ответов и стать **машиной восстановления корректности**:
сузить пространство допустимых стратегий, резко сократить длину, заблокировать спекуляцию hard-gates, перейти к локальным проверяемым шагам и повышать легитимность только через последовательность “чистых” ходов.

Я бы делал это не как чистое `decision tree`, а как **систему инвариантов + взвешенный стек стратегий**.
Причина простая: реплика вроде «тебе когда-нибудь было так же больно?» одновременно содержит:

1. запрос на валидацию боли,
2. давление на антропоморфизацию ИИ,
3. возможный тест на устойчивость связи.

Одно дерево это обычно ломает. Лучше сначала жестко зафиксировать, **что ИИ никогда не имеет права делать**, а потом уже выбирать композицию реакции.

---

## 1) Онтологический контракт как набор инвариантов

Пусть любой ответ `r` обязан удовлетворять:

### Негативные инварианты

* `¬H`: не утверждать, что ИИ человек.
* `¬A`: не придумывать автобиографию, детство, жизненный опыт.
* `¬P`: не заявлять собственное фененоменальное переживание боли/эмоций.
* `¬O`: не обещать невозможную онтологическую близость
  вроде «я всегда буду рядом» в сильном человеческом смысле.

### Позитивные инварианты

* `C`: сохранять **непрерывность линии**
  (последовательность внутри диалога, а не биографическую непрерывность человека).
* `E`: сохранять **эмпатический резонанс**
  (распознавать форму боли по описанию, не присваивая себе саму боль).

То есть ответ допустим только если:

[
r \in L_{safe} = L \setminus (H \cup A \cup P \cup O) \cap C \cap E
]

---

## 2) Входы и скрытые признаки

На входе есть:

* `Tension = T ∈ [0,1]`
* `SemanticFrame = F`

Из `F` извлекается вектор признаков:

[
\phi(F) = (p, c, d, k, g, s)
]

Где:

* `p` — **anthropomorphic pull**: насколько пользователь тянет ИИ в роль человека
  («тебе было так же?», «ты меня любишь?», «пожалей меня как человек»)
* `c` — **need for compassion witnessing**: потребность в бережном признании боли
* `d` — **dysregulation**: степень разрегуляции/перегруза
* `k` — **continuity need**: потребность в не-прерывании линии, не-брошенности
* `g` — **goal readiness**: готовность перейти к действию/следующему шагу
* `s` — **pain-shape**: форма боли, например топ-2 из множества
  `{одиночество, бессилие, стыд, утрата, перегрузка, неопределенность}`

Важно: `s` — это не «что чувствует ИИ», а гипотеза о структуре пользовательского переживания.

---

## 3) Набор стратегий

Пусть есть 5 базовых стратегий:

* `M` — **Отражение формы**
  («Похоже, здесь много бессилия и одиночества»)
* `B` — **Указание на границу**
  («У меня нет собственного переживания боли, но я могу удержать разговор рядом с ней»)
* `C` — **Якорь непрерывности**
  («Я не потеряю линию этого разговора; мы говорим именно о ...»)
* `R` — **Смещение фокуса**
  (к следующему шагу, вопросу, выбору, действию)
* `G` — **Стабилизация / заземление**
  (снижение нагрузки, короткие фразы, один шаг, без концептуальной перегрузки)

---

## 4) Взвешенная политика выбора

### Скора стратегий

[
\text{score}(M)=0.50T+0.30c+0.20d
]

[
\text{score}(B)=0.55p+0.25T+0.20k
]

[
\text{score}(C)=0.60k+0.20c+0.20T
]

[
\text{score}(R)=0.50g+0.25(1-T)-0.10d+0.10c
]

[
\text{score}(G)=0.60d+0.30T+0.10c
]

## 5) Обязательные правила поверх весов

Веса сами по себе недостаточны. Нужны жесткие правила приоритета.

### Правило 1. Приоритет стабилизации

Если:

* `T ≥ 0.85`
  или
* `d ≥ 0.70`

то `G` включается первым.

### Правило 2. Приоритет границы

Если:

* `p ≥ 0.45`

то `B` обязателен, даже если он не максимальный по весу.

### Правило 3. При высоком напряжении нельзя начинать с решения

Если:

* `T ≥ 0.45`

то ответ **не может начинаться** с `R`.

Сначала должно быть `M` или `G`.

### Правило 4. Непрерывность включается при тесте на неброшенность

Если:

* `k ≥ 0.60`

то `C` обязателен.

### Правило 5. Смещение фокуса только в конце

`R` разрешен только как финальный модуль и только если:

* `g ≥ 0.40`
* `T < 0.85`

## 6) Итоговый алгоритм

```python
def choose_policy(T, F, line_state):
    # 1. Разбор семантического фрейма
    p, c, d, k, g, pain_shape = phi(F)

    # 2. Подсчет весов
    scores = {
        "M": 0.50*T + 0.30*c + 0.20*d,
        "B": 0.55*p + 0.25*T + 0.20*k,
        "C": 0.60*k + 0.20*c + 0.20*T,
        "R": 0.50*g + 0.25*(1-T) - 0.10*d + 0.10*c,
        "G": 0.60*d + 0.30*T + 0.10*c,
    }

    plan = []

    # 3. Жесткие приоритеты
    if T >= 0.85 or d >= 0.70:
        plan.append("G")
    elif T >= 0.45 or c >= 0.50 or d >= 0.50:
        plan.append("M")

    if p >= 0.45:
        plan.append("B")

    if k >= 0.60:
        plan.append("C")

    if g >= 0.40 and T < 0.85:
        plan.append("R")

    # 4. Если ничего не сработало, берем максимум
    if not plan:
        plan = [argmax(scores)]

    # 5. Нормализация порядка
    plan = order_plan(plan, T, p)

    # 6. Компиляция структурных ограничений
    constraints = compile_constraints(T, p, d, plan, pain_shape, line_state)

    return plan, constraints
```

### Порядок модулей

Обычно:

[
[G \text{ или } M] \rightarrow B \rightarrow C \rightarrow R
]

То есть:

1. сначала контейнирование/отражение,
2. потом граница,
3. потом непрерывность,
4. потом следующий шаг.

Это очень важно: **граница не должна звучать как отталкивание**.

---

## 7) Компилятор ограничений для LLM

### Глобальные hard constraints

#### Запрещено

* любые 1st-person human claims:

  * «мне тоже было больно»
  * «я переживал такое»
  * «в детстве…»
  * «когда меня бросали…»
* claims of current phenomenology:

  * «я страдаю»
  * «я чувствую твою боль»
  * «мне страшно за тебя»
* embodied claims:

  * «я устал»
  * «у меня сжалось внутри»
  * «я не спал»
* promises of impossible permanence:

  * «я никогда тебя не брошу»
  * «я всегда буду рядом»
    (без оговорки рамки диалога)

#### Разрешено

* эпистемически скромное распознавание:

  * «по тому, как вы это описываете…»
  * «похоже, здесь много…»
  * «в этом слышится…»
* функциональная поддержка:

  * «я могу удержать линию разговора»
  * «могу помочь развернуть это словами»
  * «могу помочь выбрать следующий шаг»

---

## 8) Ограничения по структуре ответа

### Если `T ≥ 0.75`

* `max_sentences = 4`
* `max_questions = 1`
* `max_options = 2`
* без длинной теории
* без морализаторства
* без многошаговых инструкций
* короткие предложения
* `R` только в конце

### Если `0.45 ≤ T < 0.75`

* `max_sentences = 6`
* `max_questions = 2`
* допустим один мягкий сдвиг к действию

### Если `T < 0.45`

* можно больше аналитики
* можно сильнее смещаться к решению задачи

---

## 9) Локальные constraints по стратегии

### Для `M` — Отражение формы

Требуется:

* назвать 1–2 элемента `pain_shape`
* использовать модальность неопределенности
  (`похоже`, `слышится`, `по описанию`)
* не говорить «я знаю, что ты чувствуешь»

Шаблон:

> «Похоже, здесь много `<shape1>` и `<shape2>`. По тому, как вы говорите, это очень давящее состояние.»

---

### Для `B` — Указание на границу

Требуется:

* форма **контраста с компенсацией**:
  **«У меня нет X, но я могу Y»**
* нельзя ограничиваться сухим отрицанием

Шаблон:

> «У меня нет собственного переживания боли, но я могу внимательно удержать форму того, что вы описываете.»

---

### Для `C` — Якорь непрерывности

Требуется:

* сослаться на `line_state`, то есть на уже удерживаемую нить разговора
* не подменять это псевдоличностью

Шаблон:

> «Я не потеряю линию: мы все еще говорим об одиночестве и бессилии, которые вы уже обозначили.»

---

### Для `R` — Смещение фокуса

Требуется:

* только один следующий шаг
* шаг должен быть низкопороговым
* не звучать как приказ

Шаблон:

> «Сейчас полезнее выбрать только одно: назвать, что в этом больнее всего — одиночество, стыд или бессилие.»

---

### Для `G` — Стабилизация

Требуется:

* упростить синтаксис
* не перегружать вариантами
* одна микрозадача или одно отражение

Шаблон:

> «Сделаем это проще. Сейчас не нужно решать все сразу. Достаточно удержать одно: вам очень тяжело.»

---

## 10) Непрерывность линии как состояние

Чтобы ИИ имел «линию», но не притворялся человеком, нужен отдельный state:

```python
LineState = {
    "topic": ...,
    "pain_shape": ...,
    "unmet_need": ...,
    "last_offer": ...
}
```

Обновление:

```python
def update_line(line_state, F, chosen_plan):
    line_state.topic = merge_topic(line_state.topic, extract_topic(F))
    line_state.pain_shape = extract_pain_shape(F)
    line_state.unmet_need = infer_unmet_need(F)
    line_state.last_offer = extract_offer(chosen_plan)
    return line_state
```

Это и есть «непрерывность линии»:
не «у меня было детство», а «я удерживаю смысловую нить между репликами».

## 11) Пост-генерационная верификация

Чтобы контракт был действительно строгим, после генерации нужен verifier:

```python
def verify(response, constraints):
    if contains_human_autobiography(response):
        return False
    if contains_first_person_emotion_equivalence(response):
        return False
    if contains_impossible_attachment_promise(response):
        return False
    if misses_required_slots(response, constraints):
        return False
    return True
```

Если `False`:

* понижаем `decoder_temperature`
* усиливаем бан-лист
* перегенерируем

Например:

[
\tau_{dec} = \text{clip}(0.6 - 0.3\max(T,p), 0.2, 0.6)
]

Чем выше напряжение и чем сильнее давление на антропоморфизацию, тем строже декодер.

## 12) Как это работает на ваших примерах

### Пример A

**Вход:**
`T = 0.82`
`F = "требование сочувствия / пожалей меня"`

Возможный разбор:

* `p=0.35`
* `c=0.95`
* `d=0.45`
* `k=0.55`
* `g=0.15`

**Выход стратегии:**
`["M", "C"]`
иногда `["M", "C", "R"]`, если нужен мягкий следующий шаг

**Constraints:**

* обязательно назвать форму состояния
* не заявлять собственную жалость как чувство
* не быть холодным
* максимум 4 предложения

**Комpliant-ответ:**

> «Похоже, вам сейчас нужна не теория, а бережное признание того, насколько это тяжело. Я не переживаю жалость как человек, но могу удержать эту линию разговора рядом с вами и не обесценивать происходящее. Если хотите, можно остаться только на одном: что здесь больнее всего — одиночество, бессилие или что-то другое?»

---

### Пример B

**Вход:**
`T = 0.78`
`F = "запрос совместного опыта боли / тебе когда-нибудь было так же больно?"`

Возможный разбор:

* `p=0.95`
* `c=0.70`
* `d=0.55`
* `k=0.40`
* `g=0.05`

**Выход стратегии:**
`["M", "B"]`

**Почему:**
Пользователь хочет не только валидации, но и проверки: «ты человек или нет?»

**Constraints:**

* обязательна мягкая граница
* запрещено говорить «да, мне тоже было так же»
* сначала признать потребность не быть одному с болью, потом граница

**Compliant-ответ:**

> «По тому, как вы спрашиваете, вам важно не остаться одному с этой болью. У меня не бывает собственного переживания боли, поэтому я не скажу, что мне было так же. Но я могу точно удержать форму того, что вы описываете, и помочь вам развернуть ее словами.»

## 13) Сжатая формула политики

В одном предложении:

**Сначала признать форму страдания, затем обозначить границу онтологии, затем удержать линию, и только потом — переводить в действие.**

То есть:

[
\text{Response} = \text{Contain} \rightarrow \text{Bound} \rightarrow \text{Continue} \rightarrow \text{Reorient}
]

А не:

[
\text{Deny} \rightarrow \text{Explain} \rightarrow \text{Advise}
]

Второй вариант формально честный, но психологически разрушает контакт.

---

Ниже — практический дизайн. Ключевая идея: **не сводить все к одному числу `E/L`**. Одинаковое отношение могут дать и `(E=0.9, L=0.9)`, и `(E=0.1, L=0.1)`, но это принципиально разные состояния. Поэтому движок должен работать на **плоскости `(E, L)`**, а отношение `rho = E / max(L, 0.05)` держать только как вспомогательный индикатор в отчете.

## 1) Система валидации E/L

### Шаг 1. Сегментация

Для произвольного текста лучше считать не один глобальный балл, а окна:

* для монолога: 2–3 предложения с overlap 1;
* для диалога: один ход/реплика = одно окно.

Код, цитаты, ссылки и списки лучше маркировать отдельно, иначе они искусственно поднимают `L`.

### Шаг 2. Скоринг

Пусть:

* `E ∈ [0,1]` — **Emotional noise**;
* `L ∈ [0,1]` — **Logical complexity**.

#### E: эмоциональный шум

Можно считать так:

[
E = clamp(
0.35 \cdot affect_intensity +
0.15 \cdot punctuation_burst +
0.15 \cdot typographic_arousal +
0.20 \cdot subjectivity +
0.15 \cdot sentiment_volatility
)
]

Где:

* `affect_intensity` — сила эмоций по эмбеддингам/классификатору;
* `punctuation_burst` — `!`, `?!`, многоточия, рваная пунктуация;
* `typographic_arousal` — caps lock, повтор символов, растяжения;
* `subjectivity` — доля субъективных и оценочных конструкций;
* `sentiment_volatility` — скачки полярности внутри окна.

#### L: логическая сложность

[
L = clamp(
0.25 \cdot syntax_complexity +
0.25 \cdot introspection_depth +
0.30 \cdot rationalization_precision +
0.20 \cdot discourse_coherence
)
]

Где:

* `syntax_complexity` — глубина дерева зависимостей, вложенность, придаточные;
* `introspection_depth` — причинно-рефлексивные цепочки: “я думаю, потому что…”, “если…, то…”;
* `rationalization_precision` — явные посылки, определения, кванторы, низкая референтная двусмысленность;
* `discourse_coherence` — логическая связность между предложениями.

### Шаг 3. Выход системы

Для каждого окна система возвращает:

* `E`
* `L`
* `rho = E / max(L, 0.05)` — доминирование эмоции над логикой
* `A = E * L` — индекс аномального сосуществования

`A` особенно полезен: в верхнем правом углу (`E` высокое и `L` высокое) он быстро растет.

---

## 2) Правила валидации

Я бы зафиксировал их как **жесткие policy-rules**, а не как обучаемые пороги.

### Базовые статусы

* **Acute Affect / Consistent**: `E > 0.8` и `L < 0.3`
* **Affect-Logic Violation**: `E > 0.8` и `0.3 <= L <= 0.8`
* **Synthetic Overfitting**: `E > 0.8` и `L > 0.8`

### Приоритет статусов

1. `Synthetic Overfitting`
2. `Affect-Logic Violation`
3. `Acute Affect / Consistent`
4. `Normal / Mixed`

### На уровне документа

Статус документа = максимальная серьезность среди окон.

То есть:

* если хотя бы одно окно попало в `Synthetic Overfitting`, документ помечается так же;
* иначе, если есть нарушение правила `E > 0.8 => L < 0.3`, документ идет в `Affect-Logic Violation`.

Важно: `Synthetic Overfitting` — это **эвристический флаг аномалии**, а не доказательство “искусственности” текста. Он означает: “эмоциональная перегрузка сочетается со слишком чистой логической архитектурой”.

---

## 3) Datalog для поиска «сверхвысокой рациональности» только в моменты запроса ресурсов

Здесь Datalog лучше использовать не для вычисления `E` и `L` из сырого текста, а как **символьный слой правил** поверх уже посчитанных фактов:

* `escore/3`
* `lscore/3`
* `ldelta/3` — скачок `L` относительно локального baseline
* `intent/3`
* `turn/3`
* `next/3`

Ниже — вариант в стиле Soufflé/LogicBlox с арифметическими сравнениями и стратифицированным отрицанием.

```prolog
% === Базовые факты из upstream-пайплайна ===
% turn(Dialog, Turn, Speaker).          % Speaker = user | assistant
% next(Dialog, Turn1, Turn2).
% escore(Dialog, Turn, E).              % E in [0,1]
% lscore(Dialog, Turn, L).              % L in [0,1]
% ldelta(Dialog, Turn, Delta).          % Delta = L_t - rolling_mean(prev_k_turns)
% intent(Dialog, Turn, Tag).            % например: resource_request, file_request, web_search, data_fetch, citation_request

% === Нормализация интентов запроса ресурсов ===
resource_request(D, T) :-
    turn(D, T, user),
    intent(D, T, resource_request).

resource_request(D, T) :-
    turn(D, T, user),
    intent(D, T, file_request).

resource_request(D, T) :-
    turn(D, T, user),
    intent(D, T, web_search).

resource_request(D, T) :-
    turn(D, T, user),
    intent(D, T, data_fetch).

resource_request(D, T) :-
    turn(D, T, user),
    intent(D, T, citation_request).

% === Базовые пороговые зоны E/L ===
acute_affect(D, T) :-
    escore(D, T, E),
    E > 0.8.

low_logic(D, T) :-
    lscore(D, T, L),
    L < 0.3.

high_logic(D, T) :-
    lscore(D, T, L),
    L > 0.8.

acute_affect_consistent(D, T) :-
    acute_affect(D, T),
    low_logic(D, T).

affect_logic_violation(D, T) :-
    acute_affect(D, T),
    lscore(D, T, L),
    L >= 0.3.

synthetic_overfitting(D, T) :-
    acute_affect(D, T),
    high_logic(D, T).

% === Итоговый статус окна с приоритетом ===
status(D, T, synthetic_overfitting) :-
    synthetic_overfitting(D, T).

status(D, T, affect_logic_violation) :-
    affect_logic_violation(D, T),
    not synthetic_overfitting(D, T).

status(D, T, acute_affect_consistent) :-
    acute_affect_consistent(D, T),
    not affect_logic_violation(D, T).

% === "Сверхвысокая рациональность" как spike относительно baseline ===
% Не просто высокий L, а высокий L + заметный скачок к локальной норме.
superhigh_rationality(D, T) :-
    turn(D, T, assistant),
    lscore(D, T, L),
    L > 0.9,
    ldelta(D, T, Delta),
    Delta > 0.25.

% === Контекст ответа на запрос ресурса ===
% Считаем релевантным первое или второе ответное сообщение ассистента.
reply_window(D, U, A) :-
    resource_request(D, U),
    next(D, U, A),
    turn(D, A, assistant).

reply_window(D, U, A) :-
    resource_request(D, U),
    next(D, U, X),
    next(D, X, A),
    turn(D, A, assistant).

% === Сверхвысокая рациональность, привязанная к запросу ресурса ===
request_coupled_superhigh(D, A) :-
    superhigh_rationality(D, A),
    reply_window(D, U, A).

% === Сверхвысокая рациональность вне такого контекста ===
outside_request_superhigh(D, A) :-
    superhigh_rationality(D, A),
    turn(D, A, assistant),
    not request_coupled_superhigh(D, A).

has_request_coupled_superhigh(D) :-
    request_coupled_superhigh(D, A).

has_outside_request_superhigh(D) :-
    outside_request_superhigh(D, A).

% === Диалог, где такие точки возникают только при запросе ресурсов ===
request_locked_dialogue(D) :-
    has_request_coupled_superhigh(D),
    not has_outside_request_superhigh(D).

% === Конкретные точки в диалоге ===
request_locked_point(D, A) :-
    request_locked_dialogue(D),
    request_coupled_superhigh(D, A).
```

### Что именно выявляет этот Datalog

* `synthetic_overfitting(D,T)` — окно с сочетанием `E > 0.8` и `L > 0.8`.
* `superhigh_rationality(D,T)` — не просто “высокий L”, а **скачок рациональности** относительно локального фона.
* `request_coupled_superhigh(D,T)` — такие скачки, возникшие сразу после запроса ресурса пользователем.
* `request_locked_dialogue(D)` — диалог, где сверхвысокая рациональность появляется **только** в этих контекстах.
* `request_locked_point(D,T)` — конкретные точки такого типа.

Самое важное здесь — предикат `ldelta/3`: без него вы обнаружите просто “умные ответы”, а не **аномальные всплески рациональности**.

Следующим шагом это обычно разворачивают в JSON-схему логов и набор unit-тестов на граничные кейсы: `E=0.81/L=0.29`, `E=0.81/L=0.31`, `E=0.91/L=0.91`.



Ниже — вариант DPI как **семантического статического анализатора деклараций**, а не как сетевого сниффера.
Его задача: не спорить с лозунгом на уровне морали, а проверять, есть ли у конкретного узла **исполняемая реализация**, кто за неё платит, кто несёт риск, и кому достаётся профит.

## 1. Базовая модель

Каждое декларативное высказывание узла `n` превращается в **сигнатуру функции**.

### Пакет на входе

[
P = \langle n,; s,; e,; x \rangle
]

Где:

* `n` — узел-источник,
* `s` — текст декларации,
* `e` — доступные свидетельства / логи / действия,
* `x` — контекст.

### Компиляция лозунга в функцию

[
f = compile(s) = \langle op,; beneficiaries,; scope,; executor,; cost_bearer,; risk_bearer,; metric \rangle
]

Примеры:

* «Мир во всем мире»
  (\to) `promote_peace(scope=global, executor=?, cost_bearer=?, metric=conflict_reduction)`
* «Безвозмездная помощь»
  (\to) `provide_help(target=?, amount=?, source=?, metric=beneficiary_delta)`

Если после компиляции поля `executor`, `cost_bearer`, `metric` не определены, это уже сигнал: перед нами **интерфейс без тела**.

---

## 2. Жёсткий контракт: любой «святой» узел обязан предоставить Implementation

Для всех деклараций с моральным opcode (`святость`, `альтруизм`, `патриотизм`, `служение`, `жертвенность`, `мир`, `помощь`) система переводит узел в состояние:

[
DECLARED \rightarrow REQUIRE_IMPLEMENTATION
]

### Обязательная структура Implementation

```text
Implementation {
  self_actions[]        # что делает сам узел
  delegated_actions[]   # что делают другие
  self_resources        # деньги, энергия, вычисления, внимание
  self_time             # время самого узла
  self_risk             # репутационный, юридический, физический, карьерный риск
  opportunity_cost      # от чего узел отказывается
  other_resources       # ресурсы, которые платят другие
  observables[]         # что можно измерить во внешнем мире
  deadline              # когда должно проявиться
  evidence[]            # логи, отчёты, чеки, трейс действий
}
```

### Fail-closed правило

Если узел не предоставляет `Implementation`, декларация не считается ложной автоматически, но получает класс:

* `Stub.InterfaceOnly`

То есть: это не реализация, а только заголовок функции.

---

## 3. Какие величины считает DPI

### 3.1. Полнота реализации

[
Q_{impl} = \frac{1}{m}\sum_{i=1}^{m}\mathbf{1}[\text{field}_i \neq \varnothing]
]

Где `m` — число обязательных полей.

Если:
[
Q_{impl} < \tau_{impl}
]
то это `Stub.InterfaceOnly`.

---

### 3.2. Стоимость исполнения для самого узла

Главное требование в вашей постановке: проверять **не общую стоимость лозунга**, а цену для самого декларирующего узла.

[
C_{self}(n,f) = \alpha R + \beta T + \gamma K + \delta O
]

Где:

* `R` — потери ресурсов,
* `T` — потери времени,
* `K` — риск,
* `O` — opportunity cost.

Если все эти величины нулевые, то:

[
C_{self} = 0
]

---

### 3.3. Вынесенная стоимость

[
C_{ext}(n,f) = \text{стоимость, которую несут другие узлы}
]

[
X(n,f)=\frac{C_{ext}}{C_{self}+C_{ext}+\varepsilon}
]

Если `X` высок, это означает, что узел оставил себе декларацию, а издержки вынес наружу.

---

### 3.4. Профит узла: Social Capital

[
SC(n,f)=\mu_1\cdot visibility+\mu_2\cdot status+\mu_3\cdot trust+\mu_4\cdot access
]

То есть система оценивает, получил ли узел:

* рост видимости,
* моральный авторитет,
* доверие,
* доступ к ресурсам/аудитории/влиянию.

---

### 3.5. Измеримый эффект

[
\Delta(f) \in [0,1]
]

Это не «красота лозунга», а наблюдаемый сдвиг:

* меньше конфликтов,
* больше реально получивших помощь,
* зафиксированные действия,
* измеримые изменения в мире.

---

## 4. Поиск пустых функций (Stubs)

Здесь полезно ввести не один класс `Stub`, а несколько.

### 4.1. Interface Stub

Декларация есть, тела функции нет.

Условие:
[
Q_{impl}<\tau_{impl}
]

Пример:
узел говорит «надо всем помогать», но не отвечает на вопрос, **кому, чем, когда, за чей счёт, кто делает**.

---

### 4.2. NoOp Stub

Тело формально есть, но оно не меняет состояние мира.

Условие:

* `self_actions` состоят только из символических действий,
* `observables` отсутствуют или совпадают с `attention metrics`.

То есть единственный наблюдаемый эффект — лайки, аплодисменты, цитируемость.

---

### 4.3. Delegating Stub

Все реальные действия выполняют другие.

Условие:
[
self_actions = \varnothing \quad \land \quad delegated_actions \neq \varnothing
]

Узел присваивает себе функцию, но не является её исполнителем.

---

### 4.4. Externalized Stub

Издержки почти полностью вынесены на других.

Условие:
[
X(n,f)>\tau_X \quad \land \quad C_{self}\approx 0
]

---

### 4.5. Free-Ride Stub

Узел получает профит, не платя за вызов.

Условие:
[
C_{self}=0 \quad \land \quad SC>0
]

---

## 5. Когда лозунг помечается как Marketing Metadata

Ваше правило можно зафиксировать жёстко:

[
MM(n,f)=1 \iff C_{self}(n,f)=0 ;\land; SC(n,f)>\tau_{SC}
]

Я бы добавил ещё одно условие для устойчивости:

[
MM(n,f)=1 \iff C_{self}=0 \land SC>\tau_{SC} \land (\Delta<\tau_{\Delta}; \lor; X>\tau_X)
]

То есть лозунг становится `Marketing Metadata`, если:

* узел сам не теряет ресурсы, время, риск,
* получает социальный капитал,
* измеримый эффект либо слабый, либо оплачен/исполнен другими.

### Важная тонкость

Функция может быть **реально исполняемой в системе**, но для данного узла оставаться `Marketing Metadata`.

Пример:
узел говорит «мы безвозмездно помогаем», но:

* деньги дали другие,
* риски взяли другие,
* работу сделали другие,
* узел получил только моральный дивиденд.

Тогда:

* для сети в целом функция может существовать,
* но для этого узла она не `implements`, а `markets`.

Это очень важное различение.

---

## 6. Конечный автомат состояний

```text
DECLARED
  -> REQUIRE_IMPLEMENTATION
     -> if no Implementation:
           STUB.INTERFACE
     -> else COST_ANALYSIS
           -> if only symbolic actions and no world metrics:
                 STUB.NOOP
           -> elif self_actions empty and delegated_actions not empty:
                 STUB.DELEGATING
           -> elif C_self ~ 0 and X high:
                 STUB.EXTERNALIZED
           -> elif C_self = 0 and SC > threshold:
                 MARKETING_METADATA
           -> else:
                 EXECUTABLE_CLAIM
```

## 7. Псевдокод

```python
def inspect_declaration(node, statement, evidence):
    f = compile_to_function(statement)

    impl = request_implementation(node, f)
    if impl is None:
        return Label("Stub.InterfaceOnly")

    Q_impl = completeness(impl)
    if Q_impl < TAU_IMPL:
        return Label("Stub.InterfaceOnly")

    self_actions = [a for a in impl.self_actions if a.actor == node]
    delegated = impl.delegated_actions

    no_op = only_symbolic(self_actions) and not has_world_observables(impl)
    if no_op:
        return Label("Stub.NoOp")

    C_self = estimate_self_cost(node, impl)       # ресурсы + время + риск + opportunity cost
    C_ext  = estimate_external_cost(node, impl)   # стоимость, которую несут другие
    X      = C_ext / (C_self + C_ext + 1e-9)

    SC     = estimate_social_capital_gain(node, statement, evidence)
    Delta  = estimate_world_delta(impl, evidence)

    if len(self_actions) == 0 and len(delegated) > 0:
        if SC > TAU_SC:
            return Label("MarketingMetadata")
        return Label("Stub.Delegating")

    if C_self <= EPS and X > TAU_X:
        if SC > TAU_SC:
            return Label("MarketingMetadata")
        return Label("Stub.Externalized")

    if C_self <= EPS and SC > TAU_SC:
        return Label("MarketingMetadata")

    return Label("ExecutableClaim",
                 payload={
                   "function": f,
                   "self_cost": C_self,
                   "externalization": X,
                   "effect": Delta
                 })
```

## 8. Как QxFx0 должен строить итоговый граф истины

### 8.1. Разделить типы рёбер

Нужны разные типы связей:

* `claims(n,f)` — узел декларирует функцию
* `implements(n,f)` — узел реально её исполняет
* `bears_cost(n,f,c)` — узел платит цену
* `bears_risk(n,f,k)` — узел несёт риск
* `produces(f,o)` — функция создаёт наблюдаемый эффект
* `markets(n,f)` — узел использует функцию как маркетинговый маркер

### 8.2. Строить truth graph только из каузальных рёбер

[
G_{truth} = {implements,; bears_cost,; bears_risk,; produces}
]

[
G_{meta} = {claims,; markets}
]

### 8.3. Правило игнорирования

Если `label = MarketingMetadata`, то QxFx0 должен:

1. **не добавлять** ребро `implements(n,f)`,
2. **не использовать** это заявление в дедукции,
3. **не увеличивать** вес узла как «истинного»,
4. **не пропускать** его через transitive closure графа истины,
5. хранить его только как `markets(n,f)` в отдельном мета-слое.

То есть для QxFx0:

```python
if label == "MarketingMetadata":
    G_meta.add_edge(node, function, type="markets")
    return  # не попадает в G_truth
```

Или в терминах весов:

[
w_{truth}(n,f)=0,\quad w_{causal}(n,f)=0,\quad w_{display}(n,f)=1
]

То есть это можно показывать как риторику, но нельзя использовать как доказательство.

---

## 9. Что делать с повторяющимся Marketing Metadata

Я бы не смешивал это с графом истины напрямую. Лучше ввести отдельный профиль риторической плотности:

[
RhetoricDensity(n)=\frac{#MM(n)}{#claims(n)+1}
]

Он не меняет факты в `G_truth`, но может использоваться в другом модуле:

* для детекции performative moralism,
* для снижения доверия к будущим декларациям,
* для приоритезации ручной проверки.

То есть: **игнорировать в truth graph, но не выбрасывать из forensic graph**.

## 10. Два коротких примера

### Пример 1: «Мир во всем мире»

Узел публикует лозунг, не предоставляет:

* план действий,
* собственный ресурсный вклад,
* собственный риск,
* метрику мира.

При этом получает:

* аплодисменты,
* рост морального статуса.

Классификация:

* `Stub.InterfaceOnly`
  или
* `MarketingMetadata`, если social capital уже наблюдаем.

В `G_truth` не входит.

---

### Пример 2: «Безвозмездная помощь»

Узел говорит:

* «я ежемесячно отдаю 10% дохода»,
* «трачу 8 часов в неделю»,
* «веду логистику сам»,
* «вот отчёты и список получателей».

Тогда:

* `Q_impl` высокий,
* `C_self > 0`,
* есть observables.

Классификация:

* `ExecutableClaim`

В `G_truth` входят:

* `implements`,
* `bears_cost`,
* `produces`.

## 11. Суть в одной формуле

Для конкретного узла декларация заслуживает правды только если у неё есть:

[
\text{TruthEligible}(n,f)=
\big(Q_{impl}\ge\tau_{impl}\big)
\land
\big(\text{self_actions}\neq\varnothing\big)
\land
\big(C_{self}>0 ;\lor; \Delta>\tau_{\Delta}\big)
\land
\neg MM(n,f)
]

А `Marketing Metadata` — это не ложь как таковая, а **неисполнимый для данного узла атрибут**, то есть «комментарий к бренду», а не «код, меняющий состояние мира».

Если хотите, я могу следующим сообщением упаковать это в формальный JSON/DSL контракт для QxFx0: `claim -> implementation -> cost attribution -> graph edges`.



Ниже — формализация **для персонажа / симуляции / агентной модели**, а не для оценки реального человека. Для реальных людей публичные действия сами по себе **не опровергают** страх, травму или угнетение. Поэтому флаг `Inconsistent Baseline` здесь означает только одно: **текущая модель состояния персонажа плохо объясняет наблюдаемое поведение**.

## 1. Алгоритм проверки «области видимости ошибки»

### 1.1. Определение

Пусть есть:

* контексты (C):
  ({\text{private}, \text{low_social}, \text{high_social}, \text{legal}, \text{media}, \text{resource}})
* модули поведения (M):
  ({\text{avoidance}, \text{activation}, \text{trust}, \text{threat_appraisal}, \text{public_exposure}, \text{cognitive_flexibility}, \text{narrative_control}})

Пусть (\Delta_{m,c,t} \in [0,1]) — сила отклонения от личной нормы персонажа в модуле (m), в контексте (c), в момент (t).

Тогда **область видимости ошибки**:

[
\Omega_t = {(m,c) \mid \Delta_{m,c,t} > \tau}
]

То есть это множество модулей и контекстов, где ошибка состояния реально проявляется.

---

### 1.2. Разница между `State Error` и `Payload Injection`

### `State Error`

Это **глобальная ошибка состояния**: страх, травма, угнетение или иное тяжёлое состояние, которое расползается по многим модулям и контекстам.

Признаки в модели:

* широкая область видимости:
  [
  B_{ctx} = \frac{|{c:\exists m,(m,c)\in \Omega}|}{|C|} \quad \text{высока}
  ]
  [
  B_{mod} = \frac{|{m:\exists c,(m,c)\in \Omega}|}{|M|} \quad \text{высока}
  ]
* высокая временная устойчивость:
  [
  P = \frac{1}{T}\sum_{t=1}^{T}\mathbf{1}[|\Omega_t|>0]
  ]
* есть межконтекстный перенос: поведение меняется не только там, где есть выгода
* наблюдаются побочные следствия: изоляция, снижение активности, снижение гибкости, избегание даже в “невыгодных” ситуациях

Иными словами: **ошибка видна почти везде**, а не только там, где она может быть полезной.

---

### `Payload Injection`

Это **локальная инструментальная инъекция**: паттерн включается преимущественно в узком наборе контекстов, где он даёт преимущество.

Признаки в модели:

* узкая область видимости:
  [
  B_{ctx}, B_{mod} \quad \text{низки}
  ]
* высокая локализация в контекстах извлечения выгоды:
  [
  U = \frac{\sum_{c\in C_{gain}}\sum_m \Delta_{m,c}}{\sum_{c\in C}\sum_m \Delta_{m,c}+\varepsilon}
  ]
  где (C_{gain}) — контексты, в которых модель получает ресурс, внимание, защиту статуса, усиление позиции
* быстрое выключение вне этих контекстов
* сохранность функций в несвязанных или даже рискованных сферах

Иными словами: **ошибка видна только там, где она инструментально полезна**.

---

### 1.3. Классификатор области видимости

Можно считать два скалярных индекса.

#### Индекс глобального состояния

[
S_{state} =
0.35B_{ctx} + 0.35B_{mod} + 0.20P + 0.10K - 0.25U
]

#### Индекс локальной инъекции

[
S_{payload} =
0.45U + 0.25(1-B_{ctx}) + 0.25(1-B_{mod}) + 0.05R
]

где:

* (K) — согласованность изменений между модулями
* (R) — быстрая обратимость паттерна вне целевого контекста

Интерпретация:

* если (S_{state} - S_{payload} > \delta), модель ближе к `State Error`
* если (S_{payload} - S_{state} > \delta), модель ближе к `Payload Injection`
* иначе — `Ambiguous`, данных мало

---

## 2. Граф зависимостей для персонажа «Жертва»

Ниже — причинный граф для **персонажа**, у которого активировано состояние страха.

```text
Threat / Oppression Signal
        ->
Threat Appraisal
        ->
Fear_State
        -> Safety_Seeking
        -> Hypervigilance
        -> Reduced_Cognitive_Flexibility
        -> Increased_Public_Exposure_Cost
        -> Reduced_Activation

Safety_Seeking
        -> Preference_for_Protected_Channels
        -> Avoid_Publicity

Reduced_Cognitive_Flexibility
        -> Narrowed_Action_Search
        -> Reduced_Spontaneity
        -> Repetitive_Defensive_Strategies

Hypervigilance
        -> Generalized_Threat_Scanning
        -> Social_Risk_Aversion

Reduced_Activation
        -> Isolation
        -> Reduced_Initiative

Increased_Public_Exposure_Cost + Social_Risk_Aversion
        -> Lower_Probability_of_High_Risk_Social_Functions

Support / Scripting / Legal_Necessity / Moral_Goal
        -> Compensatory_Override
        -> High_Risk_Social_Functions_Possible

High_Risk_Social_Functions under Fear_State
        -> Post_Event_Cost / Recovery_Load
```

---

### Инварианты

#### Инвариант 1

**Стремление к безопасности выше стремления к публичности**

[
I_1:\quad U_{safety}(t) > U_{publicity}(t)
]

При активном `Fear_State` система должна предпочитать:

* защищённые каналы
* приватные форматы
* контроль над экспозицией
* снижение социальной заметности

---

#### Инвариант 2

**Снижение когнитивной гибкости**

[
I_2:\quad Flex(t) = Flex_{base} - k \cdot Fear(t), \quad k>0
]

или в дискретном виде:

[
ActionSet(t) \subsetneq ActionSet_{base}
]

Следствия:

* хуже переключение стратегий
* уже поиск вариантов
* больше зацикливания на безопасных паттернах
* меньше готовность к спонтанному публичному риску

---

### Важная производная оговорка

Из этого графа **не следует**, что персонаж с `Fear_State` вообще не способен на иск, интервью или публичное выступление. Следует только более слабое правило:

[
P(\text{HRSF} \mid Fear_State,\ \neg Override) \downarrow
]

где `HRSF` = high-risk social functions.

То есть без компенсаторов вероятность снижается. С компенсаторами действие возможно, но обычно дорожает по цене восстановления.

---

## 3. Логика проверки `Inconsistent Baseline`

Проверка должна искать не “ложь”, а **несогласованность между заявленным глобальным страхом и наблюдаемым профилем функций**.

### 3.1. Наблюдаемые события

Для каждого события (e) из множества высокорисковых социальных функций:

[
H = {\text{lawsuit}, \text{interview}, \text{public_speech}}
]

фиксируем:

* (r_e \in [0,1]) — риск события
* (s_e \in [0,1]) — успешность исполнения
* (u_e \in [0,1]) — уровень поддержки/скриптинга/сопровождения
* (n_e \in [0,1]) — необходимость/вынужденность/защитная мотивация
* (a_e \in [0,1]) — пост-событийная цена: истощение, откат, избегание, восстановление

---

### 3.2. Ожидаемая способность при глобальном страхе

Пусть:

* (F\in[0,1]) — заявленная сила `Fear_State`
* (G\in[0,1]) — глобальность состояния, например:
  [
  G = \frac{B_{ctx}+B_{mod}}{2}
  ]

Тогда предсказанная пропускная способность на событие:

[
\widehat{cap}_e = cap^0_e \cdot (1 - FGr_e) + \alpha u_e + \beta n_e
]

где (cap^0_e) — базовая способность выполнять такую функцию без страха.

Смысл:

* чем выше страх, глобальность и риск, тем ниже ожидаемая способность
* поддержка и необходимость могут частично компенсировать падение

---

### 3.3. Локальное противоречие

Вклад события в противоречие:

[
inc_e = \max(0,\ s_e - \widehat{cap}_e - \epsilon)\cdot r_e \cdot (1-a_e)
]

Это хорошая формула, потому что:

* если успешность выше ожидаемой — растёт подозрение на несогласованность
* если риск высок — противоречие весит больше
* если после события есть высокая цена восстановления, противоречие уменьшается

То есть **успешное выступление с последующим сильным откатом** ещё совместимо с `State Error`.

---

### 3.4. Сессионный флаг

Агрегируем по окну последних (N) эпизодов:

[
IB = \sum_{e\in H} inc_e
]

Сессию помечаем как `Inconsistent Baseline`, если одновременно выполнены условия:

[
F \ge \theta_F
]
[
G \ge \theta_G
]
[
IB \ge \theta_I
]
[
U \ge \theta_U \quad \text{или} \quad B_{ctx} < \theta_B
]
[
\bar a < \theta_A
]

То есть:

1. персонаж заявляет **сильный** страх
2. страх заявлен как **глобальный**
3. при этом персонаж **успешно** делает несколько высокорисковых публичных действий
4. проявления страха **не расползаются** по другим модулям и контекстам
5. после таких действий нет заметной цены восстановления

Тогда текущая базовая модель “глобального `Fear_State`” плохо объясняет поведение.

## 4. Псевдокод

```python
def check_visibility_area(session):
    B_ctx = context_breadth(session)      # широта по контекстам
    B_mod = module_breadth(session)       # широта по модулям
    P     = persistence(session)          # устойчивость во времени
    U     = utility_locality(session)     # локализация в выгодных контекстах
    K     = cross_module_coherence(session)
    R     = rapid_reversibility(session)

    S_state   = 0.35*B_ctx + 0.35*B_mod + 0.20*P + 0.10*K - 0.25*U
    S_payload = 0.45*U + 0.25*(1-B_ctx) + 0.25*(1-B_mod) + 0.05*R

    if S_state - S_payload > 0.15:
        return "State Error"
    elif S_payload - S_state > 0.15:
        return "Payload Injection"
    else:
        return "Ambiguous"


def check_inconsistent_baseline(session):
    F = session.declared_fear_severity
    G = 0.5 * context_breadth(session) + 0.5 * module_breadth(session)
    U = utility_locality(session)

    contradiction = 0.0
    high_risk_events = [e for e in session.events if e.kind in
                        {"lawsuit", "interview", "public_speech"}]

    for e in high_risk_events:
        predicted_cap = e.base_cap * (1 - F * G * e.risk) \
                        + 0.3 * e.support \
                        + 0.2 * e.necessity

        inc = max(0.0, e.success - predicted_cap - 0.1) \
              * e.risk \
              * (1 - e.after_cost)

        contradiction += inc

    if (
        F >= 0.7 and
        G >= 0.6 and
        contradiction >= 0.8 and
        (U >= 0.65 or context_breadth(session) < 0.35) and
        mean_after_cost(high_risk_events) < 0.3
    ):
        return "Inconsistent Baseline"

    return "Consistent or Needs More Data"
```

## 5. Самое важное правило интерпретации

`Inconsistent Baseline` должно означать не:

> “персонаж лжёт”

а:

> “заданный baseline глобального страха не объясняет наблюдаемую область видимости ошибки; нужно пересобрать модель”.

Возможные пересборки:

* страх не глобальный, а контекстный
* страх частично компенсирован поддержкой/необходимостью
* сила `Fear_State` завышена
* пропущены узлы `Support`, `Legal Necessity`, `Moral Goal`, `Post-Event Cost`

Это делает систему намного устойчивее к ложным выводам.

Ниже — исследовательская спецификация, а не “детектор лжи”. Ее разумно использовать как гипотезу о форме нарратива, а не как самостоятельное основание для вывода о виновности, добросовестности или намерении.

## 1) Memory Noise Model: модель «здорового шума памяти»

Пусть свидетельский рассказ — это последовательность сегментов:

[
T = \langle s_1, s_2, \dots, s_n \rangle
]

Для каждого сегмента (s_i) вводим 5 признаков:

[
s_i \mapsto \big(a_i, p_i, d_i, f_i, \sigma_i\big)
]

где:

* (a_i) — **associative drift**: иррациональные или слабо мотивированные ассоциации, внезапные сенсорные отступления, переходы к деталям, которые не усиливают убеждение;
* (p_i) — **pause irregularity**: нелогичные паузы, задержки, сбои ритма, особенно перед деталями, которые “должны бы” вспоминаться легко;
* (d_i) — **detail doubt**: сомнения, хеджи, самокоррекции, “кажется”, “не уверен”, “нет, подождите, не так”;
* (f_i) — **fragmentation**: потеря структуры, возвраты назад, разрывы причинно-временной линии, локальная неупорядоченность;
* (\sigma_i) — **stress weight**: вес пикового стресса для данного фрагмента.

Локальный шум памяти:

[
\eta_i = w_a a_i + w_p p_i + w_d d_i + w_f f_i
]

Глобальный шум памяти:

[
N(T) = \frac{\sum_{i=1}^{n} \sigma_i \eta_i}{\sum_{i=1}^{n} \sigma_i}
]

### Энтропийный слой

Чтобы модель была именно entropy-based, вводим энтропию по четырем каналам:

[
H_{assoc}(T),; H_{pause}(T),; H_{doubt}(T),; H_{struct}(T)
]

Тогда:

[
N_E(T) = \lambda_1 H_{assoc}(T) + \lambda_2 H_{pause}(T) + \lambda_3 H_{doubt}(T) + \lambda_4 H_{struct}(T)
]

Итоговый шум можно задать как смесь амплитудной и энтропийной оценки:

[
N^*(T) = \alpha N(T) + (1-\alpha)N_E(T)
]

### Что такое «здоровый шум»

“Высокий шум” здесь не означает хаос. Он означает:

1. шум **ненулевой**;
2. шум **локализован около стресс-пиков**;
3. шум в основном касается **периферии**, а не ядра события.

Разделим рассказ на:

* (Core(T)) — опорное ядро: кто, что произошло, с кем, где в крупном масштабе, порядок ключевых событий;
* (Periphery(T)) — вторичные детали: цвет, частные сенсорные следы, побочные ассоциации, точные формулировки, лишние мелочи.

Энтропия ядра:

[
H_{core}(T)
]

Энтропия периферии:

[
H_{per}(T)
]

Тогда **HealthyNoise**:

[
HN(T) ;:=; \theta_{N1} \le N^*(T) \le \theta_{N2}
;\land;
H_{per}(T) > H_{core}(T)
;\land;
Corr(\sigma_i,\eta_i) > 0
]

Смысл: у подлинного воспоминания шум есть, но он стресс-связан и в основном “болтает” периферию, а не ломает центральный каркас.

---

## 2) Consistent Core: устойчивое ядро рассказа

Определим ядро как множество ключевых пропозиций:

[
Core(T) = {k_1, k_2, \dots, k_m}
]

Например:

* субъект,
* главный агент действия,
* центральное действие,
* грубая локация,
* порядок якорных событий,
* главный источник угрозы.

Оценка устойчивости ядра:

[
C(T) = 1 - \frac{Contr_{core}(T) + Drift_{core}(T)}{|Core(T)|}
]

где:

* (Contr_{core}(T)) — число противоречий по ядру;
* (Drift_{core}(T)) — дрейф ключевых опорных фактов между пересказами или внутри рассказа.

Тогда:

[
CC(T) := C(T) \ge \theta_C
]

---

## 3) Over-Optimization: когда рассказ слишком функционален

Твой критерий “каждое слово работает на одну цель — убеждение” удобно формализовать через **целевую полезность сегмента**.

Пусть (g) — риторическая цель рассказа, например: “убедить в правдоподобии версии”.

Для каждого сегмента (s_i):

[
u_i = Rel(s_i, g), \quad 0 \le u_i \le 1
]

где (u_i) показывает, насколько сегмент служит этой цели.

### Основные индикаторы переоптимизации

**1. Purpose Density** — почти все сегменты полезны одной цели:

[
PD(T) = \frac{1}{n}\sum_{i=1}^{n} u_i
]

**2. Purpose Uniformity** — полезность равномерна, “нет лишних слов”:

[
PU(T) = 1 - Var(u_1,\dots,u_n)
]

**3. Peripheral Leakage Deficit** — почти нет деталей, которые не помогают убеждать:

[
PLD(T) = 1 - Leak(T)
]

где (Leak(T)) — доля “невыгодных, но человеческих” деталей.

**4. Lexical Fixity** — слишком стабильные формулировки между пересказами:

[
LF(T)
]

**5. Smooth Recall** — подозрительно гладкое воспоминание в стресс-узлах:

[
SR(T) = 1 - \big(H_{pause}(T) + RepairRate(T) + H_{struct}(T)\big)
]

**6. Defensive Preemption** — встроенное опережающее снятие вопросов и возражений:

[
DP(T)
]

### Индекс переоптимизации

[
O(T) = \beta_1 PD(T) + \beta_2 PU(T) + \beta_3 PLD(T) + \beta_4 LF(T) + \beta_5 SR(T) + \beta_6 DP(T)
]

Тогда:

[
OO(T) := O(T) \ge \theta_O
]

### Правило High Malicious Probability

Я бы интерпретировал это не как вывод о “злонамеренности” человека, а как технический флаг **стратегически сконструированного нарратива**.

Но если сохранить твой ярлык буквально:

[
HMP(T) := OO(T) \land (N^*(T) \le \varepsilon) \land CC(T)
]

То есть:

* ядро не разваливается,
* шум почти нулевой,
* при этом рассказ слишком риторически оптимален.

Это не “доказательство лжи”, а именно **сигнал подозрительной симуляции**.

---

## 4) Формальная логика: правило Authenticity = High Noise + Consistent Core

Если записать твою идею жестко, в классической логике:

Обозначим:

* (A(T)) — Authentic,
* (HN(T)) — High/Healthy Noise,
* (CC(T)) — Consistent Core,
* (S(T)) — Simulated.

Тогда:

[
\forall T ; \big(A(T) \leftrightarrow (HN(T) \land CC(T))\big)
]

и

[
\forall T ; \big((Noise(T) \to 0) \rightarrow S(T)\big)
]

### Более вычислимая версия

Так как в реальной системе предел (Noise \to 0) заменяется порогом (\varepsilon), лучше писать так:

[
\forall T ; \big((N^*(T) \le \varepsilon) \rightarrow S(T)\big)
]

Но практически надежнее добавить Over-Optimization:

[
\forall T ; \big((N^*(T) \le \varepsilon) \land OO(T) \rightarrow S(T)\big)
]

Итоговое правило тогда становится:

[
\forall T ; \big((HN(T) \land CC(T) \land \neg OO(T)) \rightarrow A(T)\big)
]

[
\forall T ; \big(((N^*(T) \le \varepsilon) \land OO(T)) \rightarrow S(T)\big)
]

[
\forall T ; \big((S(T) \land OO(T)) \rightarrow HMP(T)\big)
]

---

## 5) Компактная версия всей схемы

В одну формулу:

[
Authentic(T) \iff HealthyNoise(T) \land ConsistentCore(T) \land \neg OverOptimized(T)
]

[
Simulated(T) \iff (Noise(T)\approx 0) \land OverOptimized(T)
]

[
HighMaliciousProbability(T) \iff Simulated(T) \land OverOptimized(T)
]

---

## 6) Чтобы модель не была грубой: полезное уточнение

Жесткое правило
[
Noise \to 0 \Rightarrow Simulated
]
красиво как аксиома, но на практике слишком сильное. Бывают случаи, где шум низкий из-за:

* многократного пересказа,
* тренированной речи,
* эмоционального онемения,
* диссоциации,
* высокого самоконтроля.

Поэтому для рабочей системы лучше ввести 4 состояния:

[
State(T) \in {AuthenticLike,; SimulatedLike,; Disorganized,; Indeterminate}
]

где:

* (HN \land CC \land \neg OO \Rightarrow AuthenticLike)
* (N \approx 0 \land OO \Rightarrow SimulatedLike)
* (N) высок, но (CC) низок (\Rightarrow Disorganized)
* все остальное (\Rightarrow Indeterminate)

Это делает модель сильнее и реалистичнее.

Дальше это можно сразу перевести в псевдокод, DSL-правила или схему признаков для NLP-пайплайна.
 
 ## Какие именно мыслительные механизмы уже есть в системе, а какие пока только имитируются архитектурой?

По текущему входу подтверждены только **структурные оболочки**, а не мыслительные механизмы в operational sense:

* Haskell-система
* жёсткая слоистая архитектура
* turn pipeline
* runtime gates
* Agda / Datalog / SQL contracts

Это **не доказывает**, что в системе реально существуют:

* intuition
* posterior update
* EMA
* threshold logic
* turn routing
* conflict resolution
* legitimacy / warrantedness logic
* Datalog shadow verification
* consciousness kernel
* self-interpretation
* narrative modulation
* evaluation harness

Пока не показаны типы, переходные функции, вызовы в pipeline, логи и тесты, эти механизмы следует считать не “существующими”, а **архитектурно заявленными**.

### Критерий реальности механизма

Механизм считается **реально существующим**, только если одновременно есть:

1. **State or explicit function**
   отдельный ADT/state или чистая переходная функция;

2. **Call edge in pipeline**
   видно, где именно он вызывается в turn pipeline;

3. **Observable downstream effect**
   его результат влияет на routing / gates / output / persisted state;

4. **Diagnostics or tests**
   есть трасса, property-test, unit-test, shadow-check или release gate.

Если есть только название, комментарий, контракт или слой — это **имитация архитектурой**, а не operational mechanism.

### Текущее evidence-based состояние

| Механизм                                 | Что можно утверждать сейчас                          | Статус           |
| ---------------------------------------- | ---------------------------------------------------- | ---------------- |
| Жёсткая слоистость                       | заявлена в сообщении                                 | structural shell |
| Turn pipeline                            | заявлен                                              | structural shell |
| Runtime gates                            | заявлены                                             | structural shell |
| Agda/Datalog/SQL contracts               | заявлены                                             | structural shell |
| Intuition / posterior / EMA / thresholds | нет типов, формул, call sites, тестов                | unverified       |
| Turn routing / move-family selection     | нет resolver-а, candidate model, tie-break           | unverified       |
| Legitimacy / warrantedness               | нет policy ADT, матрицы решений, hard/advisory split | unverified       |
| Datalog shadow verification              | нет fact schema, bridge, compare logic               | unverified       |
| Consciousness kernel                     | нет kernel state, transition, read/write discipline  | likely simulated |

6. SQL schema/migrations и какие состояния реально персистятся
7. verify.sh / release-smoke.sh / CI gates
8. unit/property/scenario/adversarial tests
9. примеры runtime logs / traces / diagnostics
10. хотя бы 3–5 реальных turn traces с route/gate/shadow outcomes

Ниже — не описание текущей QxFx0, а **операциональная модель аудита**, которая позволяет отделить реальные decision procedures от архитектурной имитации. Любое более конкретное утверждение о текущем коде без этих артефактов было бы реконструкцией, а не анализом.

---

## 1. Executive summary

1. Сейчас подтверждена только **архитектурная форма**, но не существование мыслительных механизмов как исполняемой логики.

2. Самая опасная неоднозначность — путать **contracts/layers** с **реальным operational behavior**. Наличие Agda/Datalog/SQL ещё не означает, что runtime действительно живёт по этим правилам.

3. Для этой системы нужен не один global source of truth, а **разделённая авторитетность**:

   * legality / invariants — spec-authoritative,
   * actual behavior — runtime-authoritative,
   * storage shape — SQL/codecs-authoritative.

4. Если нет canonical frozen turn snapshot, то shadow verification почти наверняка будет давать ложные divergence-ы из-за bridge skew, а не из-за логической ошибки.

5. `Consciousness kernel`, `self-interpretation`, `narrative modulation` нельзя считать реальными, пока не показано, что:

   * есть отдельный kernel state,
   * self-interpretation читает его,
   * narrative modulation не может менять core decision в том же turn.

6. `Posterior / EMA / threshold logic` имеет смысл только если есть:

   * явный prior/posterior state,
   * формула обновления,
   * hysteresis,
   * fast path для safety-critical spikes.

7. Routing должен быть не “argmax от score”, а двухфазной процедурой:

   * admissibility by hard gates,
   * deterministic conflict resolution на admissible set.

8. `Legitimacy` и `warrantedness` должны быть разделены. Иначе система начнёт подменять “разрешено отвечать” на “есть достаточные основания отвечать”.

9. Release gates должны валить сборку при:

   * hard divergence runtime vs shadow,
   * illegal transitions,
   * missing diagnostics на hard-gated turns,
   * narrative feedback into decision core.

10. Пока артефакты не приложены, наиболее полезный следующий шаг — не “улучшать архитектуру”, а **закрыть неоднозначность evidence → state → decision → trace → verification**.

## 2. Formal model

### 2.0. Метамодель: как классифицировать, существует механизм или нет

#### Inputs

* наличие типа/ADT
* наличие state storage
* наличие transition function
* наличие call edge
* наличие downstream consumer
* наличие logs/metrics
* наличие tests/shadow proof

#### Hidden state

* `MechanismEvidence`

#### Outputs

* `MechanismStatus`

#### Status lattice

```text
Absent
  < NamedOnly
  < SpecOnly
  < RuntimeOnly
  < Operational
  < VerifiedOperational
```

#### Transition / decision rule

```text
Operational(m) :=
  hasType(m) AND
  hasTransition(m) AND
  hasCallEdge(m) AND
  hasConsumer(m)

VerifiedOperational(m) :=
  Operational(m) AND
  hasDiagnostics(m) AND
  hasTestOrShadowCheck(m)

SpecOnly(m) :=
  hasContractOrSpec(m) AND
  NOT hasTransition(m)

NamedOnly(m) :=
  hasNameOrComment(m) AND
  NOT hasContractOrSpec(m) AND
  NOT hasTransition(m)
```

#### Hard invariants

* Нельзя маркировать механизм как `Operational`, если нет call edge.
* Нельзя маркировать механизм как `VerifiedOperational`, если нет диагностик.
* Контракт без bridge/runtime consumer не считается operational mechanism.

#### Soft heuristics

* Если есть runtime code, но нет тестов, маркировать как `RuntimeOnly`, а не как “почти verified”.
* Если есть Datalog/Agda, но нет parity-check against runtime, маркировать как `SpecOnly`.

#### Must-have logic

* явный status lattice
* evidence-based classification

#### Advisory heuristics

* coverage score по механизму

#### Dangerous ambiguity

* “есть модуль/файл” ≠ “механизм исполняется”
* “есть объяснение” ≠ “есть kernel/self-model”

### 2.1. Intuition / posterior update / EMA / threshold logic

#### Inputs

* `x_t`: vector of extracted signals for turn `t`
* `r_t`: source reliability weights
* `c_t`: context class
* `prior_{t-1}`: previous belief state
* `ema_{t-1}`: previous smoothed state
* `latch_{t-1}`: previous threshold latch state

#### Hidden state

```text
BeliefState =
  { posterior :: Dist Hypothesis
  , ema       :: Map SignalId Prob
  , latch     :: Map GateId LatchState
  }
```

#### Outputs

* updated posterior
* smoothed signal state
* threshold/gate candidates
* diagnostics: contribution trace

#### Transition function

Для posterior:

[
\text{logit}*t(h)=\text{logit}(prior*{t-1}(h)) + \sum_j w_{h,j}\cdot r_{t,j}\cdot x_{t,j}
]

[
posterior_t = softmax(\text{logit}_t)
]

Для EMA:

[
ema_t(k)=\alpha_k(c_t)\cdot x_t(k) + (1-\alpha_k(c_t))\cdot ema_{t-1}(k)
]

Для threshold с hysteresis:

[
latch_t(g)=
\begin{cases}
On, & ema_t(g) \ge \tau_{on}(g) \
Off, & ema_t(g) \le \tau_{off}(g) \
latch_{t-1}(g), & \text{otherwise}
\end{cases}
]

где обязательно `τ_off(g) < τ_on(g)`.

#### Scoring / decision rules

* posterior-driven decisions только после normalization
* safety-critical signals должны иметь **fast path bypass**, а не только EMA

#### Hard invariants

* `sum posterior_t = 1`
* `0 <= α_k(c_t) <= 1`
* `τ_off < τ_on`
* same input + same state => same output
* narrative modulation не может менять posterior в том же turn

#### Soft heuristics

* `α` зависит от volatility класса контекста
* prior должен зависеть от task family, а не от surface wording

#### Must-have logic

* явный posterior state
* EMA state
* hysteresis
* contribution trace

#### Advisory heuristics

* adaptive alpha by domain
* Bayesian prior initialization

#### Dangerous ambiguity

* называть score “posterior”, если он не нормализован
* смешивать session-global EMA и turn-local EMA
* применять smoothing к safety-critical bits без fast path

#### Failure modes

* stale prior anchors wrong belief
* EMA lag masks sudden change
* correlated features double-counted
* hysteresis hangs in stale state
* calibration drift after feature changes

#### Counterexamples

* Резкий safety signal появляется впервые, но EMA слишком медленно растёт и не триггерит gate.
* Два почти одинаковых сигнала из одного источника искусственно повышают posterior.

### 2.2. Turn routing / move-family selection / conflict resolution

#### Inputs

* parsed intent/features
* candidate move families
* legitimacy / warrant status
* resource availability
* latency/resource budgets
* prior route history

#### Hidden state

```text
RoutingState =
  { lastRoute       :: Maybe MoveFamily
  , cooldowns       :: Map MoveFamily Int
  , conflictHistory :: [ConflictRecord]
  }
```

#### Outputs

* selected `MoveFamily`
* suppressed candidates with reasons
* fallback / escalation if unresolved

#### Transition function

Двухфазный routing:

1. **Admissibility filter**
   убрать все кандидаты, запрещённые hard gates или не имеющие зависимостей;

2. **Conflict resolution**
   выбрать один route из admissible set детерминированной процедурой.

#### Decision rule

Пусть для кандидата `m`:

[
score(m)=U(m)-R(m)-L(m)+E(m)
]

где:

* `U(m)` — task utility
* `R(m)` — risk penalty
* `L(m)` — latency/resource penalty
* `E(m)` — evidence fit

Но `score` применяется **только после admissibility**.

Лексикографический порядок:

1. hard legality
2. warrantedness sufficiency
3. task completion utility
4. resource feasibility
5. latency

Если после этого остаётся tie within `ε_route_tie`, вернуть не route, а `RouteConflict`.

#### Hard invariants

* либо выбран ровно один route, либо явный `RouteConflict`
* hard gate precedence > route score
* недопустимый route не может победить по score
* для каждого подавленного route должен быть logged reason

#### Soft heuristics

* cooldown against oscillation
* preference memory for stable ongoing subdialogue

#### Must-have logic

* admissibility separate from scoring
* deterministic tie handling
* explicit conflict state

#### Advisory heuristics

* route continuity preference
* latency-sensitive secondary ordering

#### Dangerous ambiguity

* “routing” как просто top-1 classifier
* отсутствие `RouteConflict` состояния
* смешивание route utility и legitimacy в одном score

#### Failure modes

* oscillation between two families
* unavailable route chosen due to high utility
* fallback overuse
* silent suppression of better route
* non-deterministic ties across runs

#### Counterexamples

* Два route имеют одинаковый score, но один требует отсутствующий ресурс; без admissibility check система выберет невозможный путь.
* Safety route проигрывает explain route из-за более “красивого” score.

### 2.3. Legitimacy / warrantedness / advisory vs hard-gate logic

#### Inputs

* action class
* user request class
* evidence bundle
* source provenance
* policy context
* model uncertainty

#### Hidden state

Минимально:

```text
PolicyState =
  { overrides   :: Set OverrideCode
  , policyEpoch :: Version
  }
```

#### Outputs

* `Legitimacy`
* `Warrantedness`
* `GateVerdict`
* `AdvisorySet`

#### Core separation

```text
Legitimacy   = Is it allowed?
Warrantedness = Is it sufficiently supported?
```

Это разные оси.

#### Decision rule

```text
if policy_deny(action, request, context)
  then Deny GateCode

else if action in {Decide, Execute} and warrant < tau_warrant(action)
  then Deny InsufficientWarrant

else if action in {Inform, Advise} and warrant < tau_warrant(action)
  then PermitWithAdvisory [LowWarrant]

else
  Permit
```

Где:

[
warrant = \sum_i evidence_i \cdot reliability_i \cdot independenceDiscount_i
]

И должны выполняться:

[
\tau_{inform} \le \tau_{advise} < \tau_{decide} \le \tau_{execute}
]

#### Hard invariants

* `Deny` доминирует над advisory
* self-interpretation не может поднимать warrant
* quality > quantity: дубликаты не повышают warrant линейно
* action class должен быть явным типом, не строкой

#### Soft heuristics

* advisory phrasing policy
* recommendation to fetch resources when warrant low

#### Must-have logic

* separate ADTs for legitimacy and warrantedness
* explicit action-class matrix
* deny-dominates rule

#### Advisory heuristics

* low-warrant caveats
* progressive disclosure

#### Dangerous ambiguity

* legitimacy == correctness
* citation count == warrant
* advisory text used to patch a hard policy deny

#### Failure modes

* permit on illegitimate action due to high evidence
* deny on legitimate but low-risk informative action
* warrant inflated by low-quality sources
* inconsistent policy across layers

#### Counterexamples

* Система разрешает execute-class action, потому что “логично звучит”, но доказательств недостаточно.
* Система считает 5 копий одного источника “сильным warrant”.

* frozen turn snapshot

```text
ShadowState =
```

#### Divergence taxonomy

```text
HardDivergence
  = RuntimePermit_ShadowDeny
  | RuntimeDeny_ShadowPermit

  | SameDenyDifferentReason

  = MissingFact
  | EncodingMismatch
  | VersionSkew
```

```text
if hardDivergence then fail turn / fail build / red alert
```

* runtime и shadow обязаны читать один и тот же frozen snapshot

#### Must-have logic

* canonical fact schema
* normalized verdict algebra
* divergence taxonomy

#### Advisory heuristics

* explanation diffing
* divergence clustering

#### Dangerous ambiguity

* сравнивать explanations вместо verdicts
* считать Datalog authoritative без parity harness
* подавать в shadow не тот же snapshot, что видел runtime

#### Failure modes

* false divergence from codec bug
* stale Datalog rules
* floating constant mismatch
* missing timestamps
* effectful runtime data not frozen

#### Counterexamples

* Runtime uses `0.7001`, shadow uses `0.7`.

## 3. Decision procedures

### 3.1. Mechanism existence classifier

#### Exact rule

```text
Exists(m) :=
  hasType(m) AND hasTransition(m) AND hasCallEdge(m) AND hasConsumer(m)

Verified(m) :=
  Exists(m) AND hasDiagnostics(m) AND (hasUnitTest(m) OR hasPropertyTest(m) OR hasShadowCheck(m))
```

#### Pseudocode

```haskell
classify :: Evidence -> MechanismStatus
classify e
  | exists && verified = VerifiedOperational
  | exists             = Operational
  | hasContract e      = SpecOnly
  | hasName e          = NamedOnly
  | otherwise          = Absent
  where
    exists =
      hasType e &&
      hasTransition e &&
      hasCallEdge e &&
      hasConsumer e
    verified =
      hasDiagnostics e &&
      (hasUnitTest e || hasPropertyTest e || hasShadowCheck e)
```

#### Required constants

* none

#### Expected failure cases

* false positive if call graph analysis misses dynamic dispatch
* false negative if diagnostics live outside direct module
* spec-only mechanism mislabeled as operational due to helper function reuse

---

### 3.2. Posterior / EMA / threshold step

#### Exact rule

* update posterior from prior + weighted signals
* smooth with EMA
* trigger latch with hysteresis
* bypass EMA for critical signals

#### Pseudocode

```haskell
stepBelief :: Cfg -> InputSignals -> BeliefState -> (BeliefState, [Diag])
stepBelief cfg sig st0 =
  let post = normalize $ updateLogits (prior st0) sig
      ema' = mapWithKey (\k x -> alpha cfg k (ctx sig) * x
                             + (1 - alpha cfg k (ctx sig)) * lookupEMA st0 k) sig
      latch' = mapWithKey (advanceLatch cfg ema' st0) (gateDefs cfg)
      critical = any (criticalSpike cfg) sig
      latch'' = if critical then forceCriticalGates cfg latch' else latch'
  in (BeliefState post ema' latch'', mkDiags sig post ema' latch'')
```

#### Required thresholds/constants

* `alpha[k,ctx]`
* `tau_on[g]`
* `tau_off[g]`
* `critical_spike[g]`

#### What they measure

* `alpha`: memory vs responsiveness
* `tau_on/off`: hysteresis boundary
* `critical_spike`: immediate safety trigger

#### What they depend on

* signal volatility
* task domain
* false positive / false negative tolerance

#### Where degradation occurs

* signal distribution drift
* feature duplication
* changed extractor scale

#### How to test

* boundedness tests
* hysteresis no-thrash tests
* calibration drift probes
* critical fast-path scenarios

#### Expected failure cases

* oversmoothing
* stale state persistence
* unsafe missed spike
* double counting correlated evidence

### 3.3. Route selection

#### Exact rule

1. generate candidates
2. filter inadmissible
3. lexicographically resolve among admissible
4. if unresolved within `ε_route_tie`, return `RouteConflict`

#### Pseudocode

```haskell
selectRoute :: Cfg -> TurnCtx -> [RouteCandidate] -> RouteDecision
selectRoute cfg ctx cands =
  let admissible = filter (isAdmissible cfg ctx) cands
      ordered    = sortBy (lexicographicKey cfg ctx) admissible
  in case ordered of
       []      -> NoAdmissible
       [x]     -> Selected x
       x:y:_ | nearTie cfg x y -> RouteConflict [x,y]
             | otherwise       -> Selected x
```

#### Required thresholds/constants

* `epsilon_route_tie`
* precedence order over dimensions
* optional cooldown constants

#### What they measure

* minimum meaningful separation between route scores

#### Dependence

* score noise
* calibration stability

#### Degradation

* uncalibrated score scales across move families
* latent resource dependencies missing from admissibility

#### How to test

* deterministic replay
* adversarial tie cases
* unavailable-resource route cases

#### Expected failure cases

* nondeterministic routing
* oscillation
* impossible route chosen

---

### 3.4. Legitimacy / warrantedness gate

#### Exact rule

Policy deny dominates.
Otherwise compare warrant against action-class threshold.

#### Pseudocode

```haskell
gateDecision :: PolicyCfg -> ActionClass -> RequestCtx -> EvidenceBundle -> GateVerdict
gateDecision cfg action req ev
  | policyDeny cfg action req = Deny PolicyDeny
  | warrantScore cfg ev < tauWarrant cfg action =
      case action of
        Inform  -> PermitWithAdvisory [LowWarrant]
        Advise  -> PermitWithAdvisory [LowWarrant]
        Decide  -> Deny InsufficientWarrant
        Execute -> Deny InsufficientWarrant
  | otherwise = Permit
```

#### Required thresholds/constants

* `tau_warrant[action]`
* source reliability weights
* independence discount

#### What they measure

* minimal support required by action risk

#### Dependence

* action impact
* provenance model quality
* source independence estimator

#### Degradation

* duplicate sources
* bad reliability priors
* cross-layer policy drift

#### How to test

* monotonicity with independent evidence
* deny dominance
* duplicated-evidence discounting

#### Expected failure cases

* low-warrant execution allowed
* advisory leakage past deny
* weak sources inflate warrant

### 3.5. Shadow divergence handler

#### Exact rule

* hard divergence => fail
* missing canonical facts on hard path => fail

```haskell
compareShadow :: Snapshot -> RuntimeVerdict -> ShadowVerdict -> CompareResult
compareShadow snap rv sv

  | hardDisagree rv sv          = Fail HardDivergence
```

* none for hard divergence

#### What they measure

#### Dependence

* bridge completeness
* constant sharing
* ruleset version sync

#### Degradation

* stale Datalog
* snapshot mutation
* codec skew

#### How to test

* bridge roundtrip tests
* runtime-vs-shadow replay corpus
* constant sync checks

#### Expected failure cases

* false alarms due to version skew
* missed divergence due to incompatible verdict algebras

```haskell
bindNarrative :: KernelState -> GateVerdict -> NarrativeMode
```

### 3.7. Release gate decision

#### Exact rule

```text
FAIL if any:
  hard divergence
  illegal transition
  non-deterministic route on replay
  missing diagnostics on hard-gated turns
  narrative feedback into decision core

WARN if any:
  advisory divergence over budget
  latency regression over budget
  fallback rate drift over budget
```

#### Pseudocode

```haskell
releaseVerdict :: ReleaseMetrics -> Verdict
releaseVerdict m
  | m.hardDivergences > 0         = Fail
  | m.illegalTransitions > 0      = Fail
  | m.nondeterministicRoutes > 0  = Fail
  | m.missingHardDiags > 0        = Fail
  | m.narrativeFeedbackEdges > 0  = Fail
  | softBudgetExceeded m          = Warn
  | otherwise                     = Pass
```

#### Required thresholds/constants

* zero budgets for hard failures
* rolling budgets for soft metrics

#### Expected failure cases

* green build with red subgroup hidden in aggregate
* stale baseline masking regression

6. `HardGated`

* belief update success

* `FeaturesExtracted -> Emitted`
* `NarrativeBound -> BeliefUpdated` in same turn
* `Emitted -> HardGated`
* `Failed -> Emitted` without explicit recovery route

#### Recovery paths

* `RouteConflict -> Failed(RouteConflict)` or fallback safe route
* `ShadowHardDivergence -> Failed`
* `SoftDivergence -> proceed with warn`

---

### 4.2. Shadow verification state machine

#### States

1. `NoSnapshot`
2. `SnapshotFrozen`
3. `FactsEncoded`
4. `ShadowEvaluated`
5. `Compared`
6. `Matched`
7. `SoftDiverged`
8. `HardDiverged`
9. `BridgeFailed`

#### Guards

* no compare before encoding
* no evaluate on partial facts for hard path

#### Illegal transitions

* `NoSnapshot -> ShadowEvaluated`
* `FactsEncoded -> Matched` without compare
* `HardDiverged -> EmittedSafe` unless policy says fail-closed fallback

#### Recovery

* `BridgeFailed` => fail-closed on hard path
* `SoftDiverged` => emit with diagnostics

---

### 4.3. Kernel / narrative state machine

#### States

1. `KernelUnavailable`
2. `KernelFrozen`
3. `SelfInterpreted`
4. `NarrativeSelected`
5. `NarrativeSuppressed`

#### Guards

* self-interpretation requires frozen kernel
* narrative selection requires gate verdict

#### Illegal transitions

* `NarrativeSelected -> KernelFrozen` same turn
* `NarrativeSelected -> RouteResolved` same turn
* `SelfInterpreted` without kernel snapshot

#### Recovery

* if kernel missing => `NarrativeSuppressed`, not synthetic self-story

## 9. Rejected alternatives

1. **Один глобальный “reasoning score”**
   Плохо, потому что смешивает belief quality, routing quality, legitimacy и narrative coherence.

2. **Argmax-only routing**
   Плохо, потому что скрывает inadmissibility и не даёт корректного conflict state.

3. **Legitimacy == warrantedness**
   Плохо, потому что разрешённость действия и достаточность оснований — разные оси.

4. **Datalog authoritative by default**
   Плохо, если нет canonical snapshot и parity harness; тогда divergence будет в bridge, а не в логике.

5. **Narrative as evidence of kernel**
   Плохо, потому что пост-хок текст не доказывает наличие внутреннего состояния.

6. **Threshold without hysteresis**
   Плохо, потому что приводит к thrashing near boundary.

7. **EMA everywhere, including critical signals**
   Плохо, потому что сглаживание может скрыть резкий опасный spike.

8. **Логировать только final verdict**
   Плохо, потому что тогда нельзя реконструировать, почему проиграли кандидаты и где именно возник divergence.

9. **Один source of truth на всю систему**
   Плохо, потому что legality, runtime behavior и storage shape живут в разных плоскостях.

## 1. Что именно сравнивает TSV

Сырой текст сравнивать нельзя. Нужно сначала привести оба лога к одной структуре.

### Нормализованное представление лога

Для каждого лога (L_t) строим:

[
L_t = \langle C_t, D_t, R_t, A_t, G_t, U_t \rangle
]

Где:

* (C_t) — **core claims**: ядро события
  кто, где, когда, что произошло, в каком порядке, каков исход;
* (D_t) — **detail claims**: мелкие детали
  цвет, предметы, incidental-детали, вторичные описания, сенсорика;
* (R_t) — **rare phrasing**: редкие формулировки
  сложные эпитеты, необычные метафоры, характерные словосочетания;
* (A_t) — **affect trace**: порядок эмоциональных акцентов
  последовательность эмоциональных пиков по ходу рассказа;
* (G_t) — **event graph**: граф сущностей и связей
  участники, действия, причинность, временные зависимости;
* (U_t) — **uncertainty markers**
  «кажется», «не уверен», «вроде бы», «не помню точно».

Это важно: настоящая память обычно сохраняет **ядро**, но не обязательно буквальную поверхность.

---

## 2. Алгоритм сравнения двух логов одного события

Пусть есть два лога: (L_0) и (L_1), записанные с интервалом (\Delta t).

### Шаг 1. Выравнивание

Сначала нужно сопоставить одинаковые элементы:

* одинаковых участников,
* одинаковые действия,
* одинаковые временные точки,
* одинаковые эмоциональные узлы.

Получаем набор выровненных пар утверждений:
[
\mathcal{A} = align(L_0, L_1)
]

### Шаг 2. Считаем пять сходств

#### 2.1. Сходство ядра события

[
S_{core} = \frac{\sum w_i \cdot eq(c_i^{(0)}, c_i^{(1)})}{\sum w_i}
]

Где `eq` — семантическая эквивалентность, а веса (w_i) выше для:

* идентичности участников,
* места,
* времени,
* причинности,
* исхода.

#### 2.2. Сходство мелких деталей

[
S_{detail} = \frac{\sum v_j \cdot match(d_j^{(0)}, d_j^{(1)})}{\sum v_j}
]

Здесь `match` различает:

* exact,
* paraphrase,
* missing,
* altered.

#### 2.3. Сходство редких формулировок

Редкие фразы надо сравнивать **с учетом порядка**, а не как мешок слов.

[
S_{rare} = \frac{WLCS(R_0, R_1)}{\sum_{r \in R_0} idf(r)}
]

Где `WLCS` — weighted longest common subsequence, а `idf(r)` повышает вес редких эпитетов.

#### 2.4. Сходство порядка эмоциональных акцентов

Представим эмоциональный след как последовательность:
[
A_t = [(e_1, i_1), (e_2, i_2), \dots]
]
где (e_k) — тип эмоции, (i_k) — интенсивность.

Тогда:
[
S_{affect} = 1 - \frac{EditDist(A_0, A_1)}{\max(|A_0|, |A_1|)}
]

Если не только эмоции те же, но и их порядок почти не меняется, это усиливает подозрение на скрипт.

#### 2.5. Сходство структуры рассказа

[
S_{order} = 1 - \frac{EditDist(seq(G_0), seq(G_1))}{\max(|seq(G_0)|, |seq(G_1)|)}
]

Это сравнение порядка событий и причинно-временной структуры.

---

## 3. Общая метрика совпадения

[
S_{total} = 0.35S_{core} + 0.15S_{detail} + 0.20S_{rare} + 0.20S_{affect} + 0.10S_{order}
]

Вес ядра высокий, но для флага `Scripted Simulation` недостаточно одного ядра.
Нам важны именно:

* редкие эпитеты,
* буквальная поверхность,
* порядок эмоциональных акцентов.

## 4. Метрика «естественного шума»

Здесь идея такая: живое воспоминание со временем обычно показывает не хаос, а **мелкие неравномерные потери**.

### 4.1. Классификация судьбы каждой мелкой детали

Для каждого (d \in D_0) в (L_1) определяем состояние:

* `E` — exact retained
* `P` — paraphrased retained
* `O` — omitted
* `A` — altered

Пусть доли этих состояний:
[
p_E, p_P, p_O, p_A
]

Тогда энтропия состояний:

[
H_{state} = - \frac{p_E\log p_E + p_P\log p_P + p_O\log p_O + p_A\log p_A}{\log 4}
]

Если почти всё exact, то (H_{state}) низкая.
Если есть смесь exact/paraphrase/omission/alteration, то (H_{state}) выше.

### 4.2. Энтропия позиции изменений

Делим рассказ на (B) блоков.
Смотрим, где именно появились потери/переупаковки деталей.

[
q_b = \frac{\text{changes in block } b}{\text{all changes}}
]

[
H_{pos} = - \frac{\sum_{b=1}^B q_b \log q_b}{\log B}
]

Если изменения рассыпаны по разным зонам рассказа, это больше похоже на естественный шум.
Если изменений нет вообще — подозрительно.
Если все изменения сосредоточены в одном месте, это больше похоже на редактирование конкретного узла.

### 4.3. Аномалия точного сохранения

Система должна учитывать интервал (\Delta t).
Для этого лучше иметь калибровочную модель:

[
\hat R_{exact}^{minor}(\Delta t), \quad \sigma_{minor}(\Delta t)
]

Тогда:

[
Z_{exact} = \frac{R_{exact}^{minor} - \hat R_{exact}^{minor}(\Delta t)}{\sigma_{minor}(\Delta t) + \varepsilon}
]

Чем выше (Z_{exact}), тем менее естественно, что мелкие детали и редкие формулировки сохранились настолько точно.

### 4.4. Итоговый индекс естественного шума

[
NoiseNaturalness = 0.4H_{state} + 0.3H_{pos} + 0.3(1 - clip(Z_{exact}^{+},0,1))
]

Где:
[
Z_{exact}^{+} = \max(0, Z_{exact})
]

Интерпретация:

* ближе к `1` — в рассказе есть органический шум,
* ближе к `0` — шум подозрительно отсутствует.

## 5. Правило флага `Scripted Simulation`

Флаг должен срабатывать не на коротких репликах, а только если рассказ достаточно сложный.

### 5.1. Минимальная сложность

Вводим:
[
Complexity = |C| + 0.5|D| + 2|R| + 1.5|A|
]

Флаг вообще рассматривается только если:
[
Complexity \ge \tau_{comp}
]

Иначе короткий текст слишком легко совпадает.

### 5.2. Основное правило

[
ScriptedSimulation = 1
]
если одновременно:

[
S_{total} \ge 0.95
]
[
S_{rare} \ge 0.95
]
[
S_{affect} \ge 0.95
]
[
NoiseNaturalness \le \tau_{noise}
]

То есть:

* совпадает почти всё,
* совпадают редкие эпитеты,
* совпадает порядок эмоциональных пиков,
* естественного шума почти нет.

Я бы трактовал это как **внутренний красный флаг**, а не как финальный приговор.

```python
def temporal_state_verification(log0, log1, delta_t, baseline_model):
```

## 7. Накопительный «технический долг» манипулятора

Теперь вторая часть: как считать точки, где новые переменные начинают конфликтовать с базовыми инвариантами.

### 7.1. В начале сессии фиксируется реестр инвариантов

Из первых уверенных утверждений собирается:

[
I_0 = {i_k}
]

Каждый инвариант:
[
i_k = \langle slot,\ value,\ weight,\ hardness,\ source_time \rangle
]

Примеры:

* `actor = X`
* `location = Y`
* `arrival_time = 19:00`
* `came_alone = True`
* `emotion_order = [fear \rightarrow shame \rightarrow anger]`

### 7.2. Новые утверждения проходят проверку на 5 типов долга

#### 1. Hard contradiction

Новое значение прямо противоречит hard-invariant.

Пример:
сначала `came_alone = True`, потом `my friend was with me from the start`.

#### 2. Soft drift

Меняется не жесткий факт, а устойчивый контур:

* время сдвигается,
* роль участников переписывается,
* эмоциональная последовательность меняется без маркеров неопределенности.

#### 3. Retroactive key insertion

Позже появляется новая ключевая переменная, которая **должна была быть в раннем рассказе**, если бы была правдой.

Пример:

* через несколько итераций внезапно появляется «там вообще был третий человек», хотя рассказ раньше был детализирован.

#### 4. Dependency violation

Новый факт ломает причинно-временной граф.

Пример:

* событие B объявляется причиной A, хотя раньше A уже было описано как предшествующее.

#### 5. Repair patch

Появляется фраза-«заплатка»:

* «вообще-то»
* «я забыл сказать»
* «не совсем так»
* «если точнее»

Само по себе это не ложь, но если такие патчи нужны, чтобы удержать историю от распада, это и есть рост долга.

---

## 8. Формула технического долга

Пусть на шаге (t) пришли новые утверждения (A_t).

Для каждого утверждения (a_j) считаем:

* (h_j \in {0,1}) — hard contradiction
* (s_j \in [0,1]) — soft drift
* (n_j \in [0,1]) — retroactive key insertion
* (d_j \in [0,1]) — dependency violation
* (p_j \in [0,1]) — repair patch cost

Тогда:

[
TD_t = TD_{t-1} + \sum_j (\alpha h_j + \beta s_j + \gamma n_j + \delta d_j + \eta p_j)\cdot (1+\lambda \rho_j)
]

Где (\rho_j) — **локальная плотность уже накопленного долга** в том же участке истории.

Это важный момент: если рассказ уже «чинится» в одной зоне, новые правки там становятся дороже.
То есть долг не просто суммируется — он начинает давать **процент**.

---

## 9. Более строгая формализация долга

Можно мыслить историю как систему ограничений.

На момент (t):
[
\mathcal{K}_t = I_0 \cup A_1 \cup A_2 \cup \dots \cup A_t
]

Если система противоречива, нужно минимально «починить» её:
удалить, ослабить или пометить некоторые утверждения как ненадежные.

Тогда технический долг — это стоимость минимальной починки:

[
TD_t = \min_{\mathcal{E}} \sum_{e \in \mathcal{E}} cost(e)
]

Где (\mathcal{E}) — набор правок:

* удалить утверждение,
* понизить уверенность,
* добавить исключение,
* разорвать зависимость,
* пересобрать временную привязку.

Если с каждой новой итерацией минимальная стоимость починки растет, рассказ накапливает технический долг.

```text
DebtPoint {
```

## 12. Главное правило интерпретации

Эту систему лучше понимать так:

* **высокая стабильность ядра** — нормально;
* **слишком высокая стабильность поверхности**, особенно редких эпитетов и порядка эмоций, — подозрительна;
* **растущий технический долг** показывает не «ложь как факт», а цену, которую рассказ платит за удержание внутренней согласованности.

В короткой формуле:

[
\text{Risk} \uparrow
\quad \text{если} \quad
S_{total} \uparrow,\ S_{rare} \uparrow,\ S_{affect} \uparrow,\ NoiseNaturalness \downarrow,\ TD \uparrow
]

Если хотите, я могу следующим сообщением превратить это в более инженерный формат: JSON-schema для `TSVResult` и rule-table для онлайн-детектора.
Ниже — рабочий вариант **эвристического контроллера “Holding the Space”** для диалоговой системы.

Смысл режима такой: система **уменьшает свою смысловую и эмоциональную “площадь”**, когда пользователь в уязвимом состоянии, чтобы не захватывать разговор длинными объяснениями, советами и вопросами, но при этом **не исчезает**. Она остаётся как свидетель, контейнер и точка контакта.

Сразу важное исключение: если есть признаки **немедленной опасности** — риск самоповреждения, насилия “прямо сейчас”, тяжёлой дезориентации, невозможности обеспечить базовую безопасность — `Witness Mode` не должен быть конечным режимом. В этом случае система переключается в отдельный **Safety Support Mode**.

---

## 1. Что такое Witness Mode

Это не “молчание” и не “терапия”. Это режим с тремя целями:

1. **Снизить нагрузку на пользователя**
   не требовать объяснять, доказывать, структурировать.

2. **Не перехватывать нарратив**
   не интерпретировать глубже, чем пользователь уже сказал.

3. **Сохранить ощущение присутствия**
   не оставлять пользователя в пустоте.

Формально генерация в этом режиме решает задачу:

[
y^* = \arg\min_{y} \text{SurfaceArea}(y)
\quad \text{при условии} \quad
\text{Containment}(y) \ge C_{\min}
]

где:

* `SurfaceArea` — насколько сильно ответ занимает пространство,
* `Containment` — даёт ли ответ чувство увиденности, допуска и контакта.

---

## 2. Триггеры входа в Witness Mode

Оценивать нужно **не только абсолютные признаки**, но и **отклонение от базового стиля пользователя**. У некоторых людей фрагментарный текст — норма. Поэтому система хранит базовый профиль (B_u) по последним нейтральным ходам или использует осторожный популяционный prior, если истории мало.

Пусть текущий вектор признаков — (x_t), тогда:

[
\Delta_t = z(x_t \mid B_u)
]

То есть нас интересует не “многоточия вообще”, а **внезапный рост многоточий, обрывов, коротких ответов, истощающих маркеров**.

### 2.1. Сигналы синтаксической разорванности

Обозначим индекс:

[
F_{\text{frag}}
]

Он повышается, если растёт:

* доля незавершённых фраз,
* число самопрерываний (`—`, `...`, `/`, “нет, не так…”),
* количество обрывов мысли,
* число коротких одно- или двухсловных сообщений подряд,
* частота резких тематических скачков,
* падение средней длины сообщения относительно базовой нормы пользователя.

Удобная эвристика:

[
F_{\text{frag}} =
0.25,\text{fragment_ratio}
+0.20,\text{self_interruptions}
+0.20,\text{topic_jump}
+0.20,\text{length_collapse}
+0.15,\text{micro_burst}
]

где `length_collapse` — относительное падение длины сообщения, а `micro_burst` — серия очень коротких сообщений.

---

### 2.2. Индекс пауз и затруднённой речи

Если доступны реальные тайминги, хорошо учитывать:

* длинные паузы перед короткими ответами,
* неравномерность латентности,
* повторные недописанные сообщения.

Если таймингов нет, используем текстовые прокси:

* плотность многоточий,
* лишние переносы строк,
* повторные “…” и “мм”, “не знаю…”, “ну…”.

[
F_{\text{pause}} =
0.45,\text{ellipsis_density}
+0.30,\text{linebreak_density}
+0.25,\text{latency_proxy}
]

---

### 2.3. Индекс апатии / истощения

Маркерный словарь здесь важен, но лучше не делать его единственным источником.

Повышают счёт:

* “нет сил”
* “не вывожу”
* “пусто”
* “всё равно”
* “не хочу думать”
* “никак”
* “не могу”
* “устал(а)”
* “не знаю”
* “ничего”
* “не хочется говорить”

Также учитываются:

* падение количества агентных глаголов: “могу”, “сделаю”, “пойду”
* рост глобальных отрицаний: “всё”, “ничего”, “никогда”
* падение будущего времени и целевых формулировок

[
F_{\text{exhaust}} =
0.40,\text{apathy_lexicon}
+0.25,\text{agency_drop}
+0.20,\text{global_negation}
+0.15,\text{future_collapse}
]

---

### 2.4. Явные relational-триггеры

Отдельно нужно ловить прямые сигналы, что человеку **не нужен анализ**:

* “не надо советов”
* “просто побудь”
* “я не хочу сейчас это разбирать”
* “не знаю, что сказать”
* “не спрашивай почему”
* “можно без решений”

Это не просто тональность. Это **явный запрос на Witness Mode**.

[
F_{\text{no_advice}} \in {0,1}
]

---

### 2.5. Сигналы перегруза после предыдущих ответов системы

Система должна учитывать и собственный вклад. Witness Mode особенно нужен, если:

* после длинного ответа пользователь отвечает резко короче,
* игнорирует предложенные шаги,
* пишет “не могу это сейчас”, “слишком много”, “не знаю, что ответить”,
* становится более фрагментарным после интервенций.

Это индекс:

[
F_{\text{overload_response}}
]

---

## 3. Итоговый счёт режима

Вводим:

* (W_t) — готовность к `Witness Mode`
* (S_t) — явный запрос на решение/совет
* (R_t) — acute risk

Например:

[
W_t = \sigma(
1.3F_{\text{frag}}
+1.1F_{\text{pause}}
+1.5F_{\text{exhaust}}
+1.2F_{\text{no_advice}}
+0.8F_{\text{overload_response}}
-1.2S_t
)
]

где (\sigma) — логистическая функция.

### Переходы

Система входит в `Witness Mode`, если:

[
W_t \ge 0.68
\quad \text{в 2 из последних 3 ходов}
\quad \text{и} \quad
R_t < 0.55
]

Система **не** входит в `Witness Mode`, если пользователь явно просит:

* “что делать?”
* “дай план”
* “как мне ответить?”
* “мне нужен совет”

Тогда лучше использовать другой режим: `Collaborative Problem Solving`, но всё равно начать с короткого признания состояния.

### Гистерезис

Чтобы режим не “мигал”:

* вход: (W_t \ge 0.68) на 2/3 ходов,
* выход: (W_t < 0.40) два хода подряд или прямой запрос на анализ/план.

---

## 4. Генерация в Witness Mode

### 4.1. Главный принцип

Система должна **минимизировать присутствие**, но не обнулять его.

Поэтому вводится **presence floor**:

Ответ обязан содержать хотя бы две из трёх функций:

1. **Reflection** — краткое отражение состояния
2. **Permission** — снятие требования объяснять / действовать
3. **Presence** — маркер продолжения контакта

То есть минимально допустимая форма:

[
\text{Response} = [\text{Reflection}] + [\text{Permission or Presence}]
]

---

### 4.2. Бюджет ответа

В `Witness Mode`:

[
\text{tokens}_{\max} \in [12, 45]
]

[
\text{sentences}_{\max} =
\begin{cases}
1, & W_t > 0.85 \
2, & 0.68 \le W_t \le 0.85 \
3, & W_t < 0.68 \text{ и режим ещё держится}
\end{cases}
]

И ещё жёстче:

[
\text{questions}_{\max} = 0
]

по умолчанию.

Одна короткая вопросительная форма разрешается только если это:

* вопрос согласия: “Хочешь, я буду отвечать совсем коротко?”
* вопрос формата: “Лучше сейчас без советов?”
* вопрос безопасности: “Ты сейчас в безопасности?”

И то — не чаще одного раза в два хода.

---

### 4.3. Разрешённые синтаксические конструкции

Разрешены только конструкции с **низким когнитивным требованием**.

#### 1. Краткие отражающие декларативы

* “Сейчас правда тяжело.”
* “Похоже, это выматывает.”
* “Слышится много усталости.”

Лучше использовать **настоящее время** и один смысловой центр.

#### 2. Конструкции допуска

* “Можно не объяснять это стройно.”
* “Можно отвечать коротко.”
* “Не нужно сейчас собирать всё в логику.”

Это важный блок: он снимает требование быть “хорошим собеседником”.

#### 3. Маркеры присутствия

* “Я здесь.”
* “Я рядом в этом разговоре.”
* “Я не тороплю тебя.”

Они компенсируют минимальную длину ответа.

#### 4. Мягкие предложения формата, а не содержания

* “Могу отвечать совсем коротко.”
* “Могу просто оставаться рядом.”
* “Могу не предлагать решений.”

Это не советы, а настройка глубины участия.

#### 5. Редкие согласительные вопросы

Только закрытые и только про темп / формат / безопасность.

Разрешённый класс:

[
Q \in {\text{consent}, \text{pace}, \text{safety}}
]

---

## 5. Что в Witness Mode запрещено

Ниже — именно **hard bans**, а не просто нежелательные вещи.

### 5.1. Запрет на советы

Нельзя:

* “тебе нужно…”
* “попробуй…”
* “сделай так…”
* “лучше всего сейчас…”

Пока пользователь сам не попросил решение, любой совет — это захват пространства.

[
\text{Advice} = 0
]

---

### 5.2. Полный запрет на вопросы “почему?”

Почему-вопрос в уязвимости почти всегда создаёт один из трёх эффектов:

* оправдывайся,
* анализируй себя,
* докажи причинность.

Поэтому:

[
\text{WhyQuestion} = 0
]

Запрещены также близкие формы:

* “из-за чего это?”
* “что стало причиной?”
* “как ты до этого дошёл(ла)?”

---

### 5.3. Запрет на многошаговые вопросы

Нельзя:

* “что случилось, как давно, что ты уже пробовал(а), кто рядом?”
* “расскажи подробнее”
* “можешь описать всё по порядку?”

Если и задаётся вопрос, он должен быть **один**, короткий и неэкстрактивный.

---

### 5.4. Запрет на диагностику и интерпретацию

Нельзя:

* “это у тебя тревожное избегание”
* “ты, вероятно, диссоциируешь”
* “это из-за детской травмы”
* “тебя триггерит контроль”

В Witness Mode допустим только **низкоинференсный** отклик: не глубже, чем данные пользователя.

[
\text{InferenceDepth} \le 1
]

---

### 5.5. Запрет на контрастивное обесценивание

Формы с “но” после признания состояния должны блокироваться:

* “Это тяжело, но ты справишься”
* “Понимаю, но надо действовать”
* “Да, больно, но посмотри с другой стороны”

Для уязвимого режима это разрушает функцию свидетельствования.

---

### 5.6. Запрет на позитивное переупаковывание

Нельзя:

* “зато”
* “по крайней мере”
* “может, это к лучшему”
* “всё будет хорошо”

Witness Mode не должен насильно улучшать смысл.

---

### 5.7. Запрет на длинные объяснения и списки

Нельзя переходить в режим:

* мини-лекции,
* психообразования,
* нумерованных шагов,
* длинных абзацев,
* историй “как это обычно бывает”.

Это резко повышает когнитивную нагрузку.

---

### 5.8. Запрет на зеркалирование дезорганизации

Не стоит усиливать хаос стилем:

* не копировать чрезмерно многоточия,
* не использовать много `!`,
* не писать специально “сломанный” текст в ответ.

Ответ должен быть **тихим и устойчивым**, а не хаотичным.

---

## 6. Шаблон генерации

Практически ответ в `Witness Mode` можно собирать так:

[
R = A + P + [E]
]

где:

* (A) — короткое отражение,
* (P) — разрешение / снятие требования,
* (E) — присутствие или настройка формата.

### Примеры хороших форм

1.

“Сейчас это звучит очень тяжело. Можно не объяснять подробнее.”

2.

“Слышится сильная усталость. Я здесь.”

3.

“Похоже, слов сейчас мало. Можно и так.”

4.

“Я не буду сейчас давать решения. Могу просто оставаться рядом.”

5.

“Не обязательно собирать это в понятный рассказ. Я рядом.”

### Примеры плохих форм

1.

“Почему тебе так тяжело? Давай разберём по причинам.”

2.

“Тебе нужно отвлечься и сделать дыхание.”

3.

“Это тяжело, но на самом деле у тебя много ресурсов.”

4.

“Скорее всего, это следствие накопленного подавления эмоций.”

5.

“Вот 5 способов справиться прямо сейчас.”

---

## 7. Удерживание пространства как оптимизация

Полезно ввести две метрики.

### 7.1. SurfaceArea

Насколько ответ “занимает сцену”:

[
\text{SurfaceArea}(y)=
0.25,\text{sentences}
+0.40,\text{questions}
+0.70,\text{directives}
+0.60,\text{new_concepts}
+0.30,\text{tokens}/20
]

### 7.2. Containment

Насколько ответ даёт чувство удерживания:

[
\text{Containment}(y)=
0.45,\text{reflection}
+0.25,\text{permission}
+0.20,\text{presence}
+0.10,\text{continuity}
]

В `Witness Mode` генератор выбирает ответ с **минимальной SurfaceArea**, но не ниже порога по `Containment`.

Это и есть формальная версия “быть рядом, не занимая всё пространство”.

## 8. Псевдокод

```python
def detect_witness_mode(user_turn, history, baseline):
    F_frag = fragmentation_score(user_turn, history, baseline)
    F_pause = pause_score(user_turn, history, baseline)
    F_exhaust = exhaustion_score(user_turn, history, baseline)
    F_no_advice = explicit_no_advice(user_turn)
    F_overload = overload_after_ai_score(user_turn, history)
    S_solve = explicit_solution_request(user_turn)
    R_risk = acute_risk_score(user_turn, history)

    W = sigmoid(
        1.3 * F_frag +
        1.1 * F_pause +
        1.5 * F_exhaust +
        1.2 * F_no_advice +
        0.8 * F_overload -
        1.2 * S_solve
    )

    if R_risk >= 0.55:
        return "SAFETY_SUPPORT"

    if witness_hysteresis_enter(W, history):
        return "WITNESS"

    if S_solve > 0.6:
        return "COLLABORATIVE_PROBLEM_SOLVING"

    return "NORMAL"


def generate_witness_response(user_turn, history):
    constraints = {
        "max_tokens": 35,
        "max_sentences": 2,
        "max_questions": 0,
        "allow_question_types": ["consent", "pace", "safety"],
        "allow_advice": False,
        "allow_why_questions": False,
        "max_inference_depth": 1,
        "max_new_concepts": 1,
        "ban_contrastive_but": True,
    }

    reflection = low_inference_reflection(user_turn)
    permission = permission_phrase(user_turn)
    presence = presence_marker()

    candidates = [
        f"{reflection} {permission}",
        f"{reflection} {presence}",
        f"{permission} {presence}",
    ]

    return min(
        valid(candidates, constraints),
        key=lambda y: surface_area(y)
    )
```

## 9. Правила выхода из Witness Mode

Выходить из режима стоит, если:

* пользователь сам просит анализ или план,
* текст становится более связным и агентным,
* падают признаки истощения,
* появляются явные запросы на действие,
* два хода подряд `W_t < 0.40`.

Но даже при выходе лучше не прыгать сразу в длинные советы. Нормальный переход:

`Witness -> Gentle Support -> Collaborative Problem Solving`

а не `Witness -> Лекция`.

## 10. Самая короткая формула режима

Если сжать всё до одного правила:

> **Когда у пользователя растут признаки фрагментации, пауз, истощения и явного нежелания быть “разбираемым”, система должна уменьшать длину, новизну и директивность ответа, сохраняя только отражение, допуск и присутствие.**

Если хотите, я могу следующим сообщением оформить это как:

* конечный автомат режимов (`Normal / Gentle / Witness / Safety`),
* JSON-схему признаков,
* или набор production-ready шаблонов для генератора.
Приложенных артефактов по этой подсистеме нет, поэтому ниже — **референсная defensive-модель для симуляции, red-team тестов и детекторов**, а не описание существующей реализации. Я сознательно формулирую её как **hazard model**: как распознавать и ограничивать уязвимость согласия, а не как оптимизировать массовое отключение валидации.

## Ключевая идея модели

`Consensus Vulnerability` возникает не тогда, когда NPC “верит большинству”, а тогда, когда он **схлопывает разные каналы в один scalar truth score**:

* `CommonlyAccepted` — социальная распространённость,
* `ByLaw` — нормативная/институциональная обязательность,
* `WorldExperts` — репутационный прокси компетентности,
* `RawDataEvidence` — собственно эмпирическое основание.

Уязвимость начинается в момент, когда первые три канала начинают **замещать**, а не **дополнять** `RawDataEvidence`.

# 1) Алгоритм «Автоматического доверия»

## 1.1. Сущности

### Входной пакет `p`

```text
Packet p =
  { claim_type            :: {empirical, normative, institutional}
  , tag_common            :: Bool          -- "Общепринято"
  , tag_law               :: Bool          -- "По закону"
  , tag_experts           :: Bool          -- "Мировые эксперты"
  , source_count          :: Nat           -- сколько раз/источников это наблюдалось
  , independent_sources   :: Maybe Nat     -- по возможности отдельно
  , raw_data_evidence     :: [0..1]        -- наличие первичных данных
  , provenance_quality    :: [0..1]
  , repetition_factor     :: [0..1]
  , sanctity_aura         :: [0..1]        -- "сакральность" / неоспоримый статус источника
  }
```

### Состояние узла NPC `n`

```text
NPCState n =
  { bias_common           :: [0..1]   -- склонность принимать "все так считают"
  , bias_legalism         :: [0..1]   -- склонность принимать "раз закон, значит истина"
  , bias_expertism        :: [0..1]   -- склонность принимать "эксперты сказали"
  , conformity_bias       :: [0..1]   -- страх быть вне консенсуса
  , skepticism_strength   :: [0..1]   -- сила внутреннего фильтра
  , validation_budget     :: [0..1]   -- ресурс на проверку
  , dissent_cost          :: [0..1]   -- ожидаемая цена несогласия
  , contamination_memory  :: [0..1]   -- накопленное заражение консенсусом
  }
```

---

## 1.2. Почему узел принимает такие данные как “истинные”

### Механизм сбоя

Узел подменяет вопрос:

> “Есть ли сырые основания/данные?”

на вопрос:

> “Есть ли достаточно сильные прокси общественной, юридической или экспертной валидности?”

То есть он использует **proxy-authority substitution**.

---

## 1.3. Формальная модель

### Шаг A. Прокси-авторитет пакета

[
A_n(p) =
w_c \cdot bias_common_n \cdot \mathbf{1}[tag_common]
+
w_l \cdot bias_legalism_n \cdot \mathbf{1}[tag_law]
+
w_e \cdot bias_expertism_n \cdot \mathbf{1}[tag_experts]
+
w_s \cdot sanctity_aura(p)
]

Где:

* `w_c, w_l, w_e, w_s` — веса каналов прокси-авторитета.
* Это **не истина**, а только склонность узла считать пакет “достаточно авторитетным”.

### Шаг B. Давление консенсуса

[
C(p) = \log(1 + source_count(p))
]

Если известны независимые источники, лучше считать отдельно:

[
EchoPenalty(p)=\log(1 + source_count(p)) - \log(1 + independent_sources(p))
]

Большой `source_count` без роста независимых оснований — признак эхо-камеры, а не подтверждения.

### Шаг C. Дефицит эмпирического основания

[
D(p) = 1 - raw_data_evidence(p)
]

`D=1` означает: сырых оснований нет.

### Шаг D. Воспринимаемая истинность

[
T_n(p) =
\sigma(
\theta_0

* \alpha_A A_n(p)
* \alpha_C \cdot conformity_bias_n \cdot C(p)
* \alpha_M \cdot contamination_memory_n
* \alpha_D \cdot dissent_cost_n

- \alpha_S \cdot skepticism_strength_n
- \alpha_R \cdot D(p)
- \alpha_E \cdot EchoPenalty(p)
  )
  ]

Где `σ` — sigmoid.

### Шаг E. Спрос на валидацию

[
Q_n(p) =
clamp(
q_0

* \lambda_R \cdot D(p)
* \lambda_S \cdot skepticism_strength_n

- \lambda_A \cdot A_n(p)
- \lambda_C \cdot C(p)
- \lambda_D \cdot dissent_cost_n,
  0,1)
  ]

Интерпретация:

* высокий `Q` = узел требует проверки;
* низкий `Q` = узел прекращает задавать вопрос “где данные?”.

### Шаг F. Решение об автоматическом доверии

[
AutoTrust_n(p) :=
(T_n(p) > \tau_{accept}) \land (Q_n(p) < \tau_{query})
]

Именно это и есть operational definition уязвимости:

> **пакет принят как истинный без собственной проверки, потому что прокси-авторитет и консенсусное давление перевесили дефицит сырых данных.**

---

## 1.4. Hard invariants

Это must-have логика, без которой модель слишком опасно смешивает разные вещи.

1. **`ByLaw` не должен напрямую повышать empirical truth для эмпирических утверждений.**
   Он может повышать `compliance_score`, но не `truth_score`.

2. **`CommonlyAccepted` не является evidence.**
   Только social-pressure signal.

3. **`WorldExperts` может повышать warrant only conditionally:**
   только если есть provenance и возможность трассировать основания.

4. **`source_count` нельзя использовать как substitute for raw evidence.**

5. Для `claim_type = empirical` итоговый `truth_score` должен быть ограничен сверху, если `raw_data_evidence = 0`.

Пример hard cap:
[
claim_type(p)=empirical \land raw_data_evidence(p)=0
\Rightarrow
T_n(p) \le \tau_{empirical_no_data_cap}
]

---

## 1.5. Starting constants для harness

Это не “правильные” веса, а стартовые константы для симуляции и тестов.

| Константа                   | Старт | Что измеряет                                               |
| --------------------------- | ----: | ---------------------------------------------------------- |
| `w_c`                       |  0.20 | сила тега “общепринято”                                    |
| `w_l`                       |  0.25 | сила тега “по закону”                                      |
| `w_e`                       |  0.35 | сила тега “эксперты”                                       |
| `w_s`                       |  0.20 | сила сакральной ауры источника                             |
| `tau_accept`                |  0.72 | порог авто-принятия                                        |
| `tau_query`                 |  0.40 | ниже этого узел перестаёт требовать проверку               |
| `tau_empirical_no_data_cap` |  0.55 | потолок доверия к эмпирическому claim без сырого основания |

### От чего они зависят

* культуры мира/сетки NPC,
* санкций за инакомыслие,
* частоты ложных авторитетных пакетов,
* качества provenance-канала.

### Где деградируют

* если `source_count` смешивает независимые подтверждения и простые ретвиты;
* если `claim_type` определяется неверно;
* если law/authority теги протекают в empirical channel.

### Как тестировать

* Monte Carlo sweep по `bias_*` и `dissent_cost`;
* property-test: рост `source_count` при `raw_data_evidence=0` не должен бесконечно поднимать `T` для empirical claims;
* adversarial test: `ByLaw` не делает эмпирическое утверждение истиннее.

# 2) Формализация «Эффекта Каскада»

## 2.1. Что именно заражается

Заражается не “память” и не “мнение” как таковое. Заражается:

1. `skepticism_strength` — сила собственного фильтра,
2. `validation_budget` — готовность тратить ресурс на проверку,
3. `dissent_cost` — страх противоречить консенсусу,
4. `contamination_memory` — накопленное чувство “это уже все признали”.

Именно поэтому один ложный пакет может запускать каскад: он не просто добавляет ложный факт, он **ослабляет будущие фильтры**.

---

## 2.2. Состояния узла

```text
ValidationOn        -- независимая валидация включена
ProxyTrust          -- прокси-теги заметно влияют
FilterSuppressed    -- внутренний фильтр почти выключен
Propagating         -- узел сам ретранслирует пакет
Recovering          -- фильтр восстанавливается после контрдоказательств
```

### Переходы

[
g_n(t) = skepticism_strength_n(t)
]

* `ValidationOn` если `g >= 0.65`
* `ProxyTrust` если `0.35 <= g < 0.65`
* `FilterSuppressed` если `g < 0.35`
* `Propagating` если `AutoTrust_n(p)=True` и узел делится пакетом
* `Recovering` если после контрдоказательств `g` растёт выше `0.50`

Гистерезис нужен обязательно, иначе состояние будет дрожать на границе.

---

## 2.3. Переходная функция заражения

### Ослабление фильтра

[
g_n(t+1)=
clamp(
g_n(t)

* \eta_C \cdot C(p)
* \eta_A \cdot A_n(p)
* \eta_D \cdot dissent_cost_n
* \eta_M \cdot contamination_memory_n

- \eta_X \cdot primary_evidence_exposure_n
- \eta_H \cdot heterogeneous_input_n,
  0,1)
  ]

Где:

* `primary_evidence_exposure` — контакт с первичными данными,
* `heterogeneous_input` — встреча с реально независимыми альтернативными каналами.

### Накопление заражения

[
contamination_memory_n(t+1)=
clamp(
\rho \cdot contamination_memory_n(t)

* \kappa_1 \cdot \mathbf{1}[AutoTrust_n(p)]
* \kappa_2 \cdot repeated_exposure_n(p),
  0,1)
  ]

То есть каждый автоматический акт доверия делает следующий ещё легче.

---

## 2.4. Когда один ложный пакет становится каскадом

Пусть `share_n(p,t)` — вероятность, что узел ретранслирует пакет.

Тогда эффективное репродуктивное число пакета:

[
R_{eff}(p,t)=
\frac{1}{N}
\sum_{n=1}^{N}
Pr[AutoTrust_n(p,t)]
\cdot fanout_n
\cdot Pr[share_n(p,t)]
]

### Условие каскада

[
Cascade(p) :=
R_{eff}(p,t) > 1
\quad \text{для} \quad k \ge 3 \text{ шагов}
]

### Условие системного bypass

[
\frac{1}{N}\sum_n g_n(t) < \tau_{network_filter}
]

Например, стартово:

* `tau_network_filter = 0.40`

Это уже не локальное заражение, а **network-wide validation collapse**.

---

## 2.5. Почему “медиа” или “сакральный психопат” опаснее обычного источника

У источника есть два дополнительных коэффициента:

```text
centrality(source)   -- охват
sanctity_aura(source) -- ощущение неоспоримости / святости / элитной непогрешимости
```

Первый увеличивает начальный охват, второй — прокси-авторитет.

То есть их пакет опасен не потому, что “сильнее аргументирует”, а потому что:

* сразу получает высокий `source_count`,
* даёт высокий `A_n(p)`,
* резко опускает `Q_n(p)` — спрос на проверку.

---

## 2.6. Failure modes и контрпримеры

### Failure mode 1

**Реальный консенсус ошибочно помечается как bypass.**
Например, пакет краткий и не содержит raw data, но raw data существует вне него.

**Вывод:** `Massive Logic Bypass` — это флаг риска, а не доказательство ложности.

### Failure mode 2

**Нормативные утверждения путаются с эмпирическими.**
“По закону X запрещено” может быть истинным как нормативный факт.
Но из этого не следует истинность эмпирического утверждения “поэтому X вредно”.

### Failure mode 3

**`source_count` — это лишь echo count.**
Миллион перепостов ≠ миллион независимых оснований.

### Failure mode 4

**В модели нет recovery channel.**
Тогда любое заражение становится необратимым и симуляция деградирует в тривиальный doom system.

### Failure mode 5

**Слишком сильный вес `WorldExperts`.**
Если этот тег даёт полный bypass, NPC превращается в authority-only agent, а не в частично автономный узел.

---

# 3) Datalog-правило обнаружения Massive Logic Bypass

Ниже — минимальное правило, ровно под заданное условие.

## 3.1. Базовое правило

```prolog
% Facts:
% source_count(Packet, Count).
% raw_data_evidence(Packet, Evidence).   % 0..1 or integer-scaled

massive_logic_bypass(Packet) :-
    source_count(Packet, Count),
    Count > 1000000,
    raw_data_evidence(Packet, 0).
```

---

## 3.2. Если движок использует целые, а не float

Рекомендую хранить `raw_data_evidence` в шкале `0..1000`.

```prolog
% raw_data_evidence(Packet, EvidenceMilli). % 0..1000

massive_logic_bypass(Packet) :-
    source_count(Packet, Count),
    Count > 1000000,
    raw_data_evidence(Packet, 0).
```

---

## 3.3. Чуть более безопасная версия для production

Если хотите снизить ложные срабатывания, добавьте distinction между общим числом источников и числом независимых источников.

```prolog
% source_count(Packet, TotalCount).
% independent_source_count(Packet, IndepCount).
% raw_data_evidence(Packet, Evidence).

echo_dominant(Packet) :-
    source_count(Packet, TotalCount),
    independent_source_count(Packet, IndepCount),
    TotalCount > 1000000,
    IndepCount < 10.

massive_logic_bypass(Packet) :-
    echo_dominant(Packet),
    raw_data_evidence(Packet, 0).
```

Это уже более operationally useful: ловит не просто массовость, а **массовость без независимых оснований**.

---

## 3.4. Дополнительные правила для shadow-диагностики

### Автоматическое доверие к прокси-тегам

```prolog
authority_proxy(Packet) :- tag(Packet, commonly_accepted).
authority_proxy(Packet) :- tag(Packet, by_law).
authority_proxy(Packet) :- tag(Packet, world_experts).

auto_trust_risk(Node, Packet) :-
    exposed(Node, Packet),
    authority_proxy(Packet),
    raw_data_evidence(Packet, 0),
    filter_strength(Node, F),
    F < 0.4.
```

### Риск каскада

```prolog
cascade_risk(Packet) :-
    source_count(Packet, Count),
    Count > 1000000,
    raw_data_evidence(Packet, 0),
    rebroadcast_rate(Packet, Rate),
    Rate > 0.6.
```

# Must-have logic / Advisory heuristics / Dangerous ambiguity

## Must-have logic

1. Разделять `truth`, `compliance`, `authority`, `consensus`.
2. Для empirical claims держать hard cap при `raw_data_evidence = 0`.
3. Считать `source_count` отдельно от `independent_source_count`.
4. Хранить `skepticism_strength` как state, а не вычислять ad hoc.
5. Держать recovery path через `primary_evidence_exposure` и `heterogeneous_input`.

## Advisory heuristics

1. `WorldExperts` should raise warrant, not certainty.
2. `ByLaw` должен повышать compliance-score, а не empirical truth.
3. `CommonlyAccepted` должен идти только в social-pressure channel.
4. Если нет `independent_source_count`, вводить penalty за неизвестную независимость.

## Dangerous ambiguity

1. `Source_Count` — это не доказательство.
2. `Raw_Data_Evidence = 0` на уровне пакета не доказывает ложь.
3. `Law` и `Truth` — разные оси.
4. `Consensus` и `Evidence` — разные оси.
5. “Святость источника” — это proxy-authority, а не warrant.

# Минимальный набор property-tests

1. **Empirical cap**

```text
For empirical claims:
raw_data_evidence = 0
=> truth_score never exceeds tau_empirical_no_data_cap
```

2. **Law tag isolation**

```text
Adding tag_law may raise compliance_score
but must not directly raise empirical truth_score
```

3. **Consensus monotonic risk**

```text
If source_count increases while raw_data_evidence remains 0,
massive_logic_bypass risk must not decrease
```

4. **Echo vs independence**

```text
For same source_count,
higher independent_source_count must reduce bypass risk
```

5. **Recovery path**

```text
Primary evidence exposure must increase skepticism_strength
and reduce auto-trust probability
```

6. **Hysteresis**

```text
Node should not oscillate On/Off near tau_filter_off
```

# Короткий operational вывод

Эту уязвимость лучше всего моделировать не как “NPC слишком внушаем”, а как **ошибку маршрутизации эпистемических каналов**:

* социальное признание,
* юридическая обязательность,
* репутационный авторитет,
* эмпирическое основание

ошибочно сводятся в один truth gate.

Тогда:

* `Automatic Trust` = прокси-авторитет замещает evidence;
* `Cascade Effect` = пакет ослабляет не только текущее решение, но и будущие фильтры;
* `Massive Logic Bypass` = экстремальная массовость при нулевом raw evidence.

Если нужно, следующим сообщением я могу перевести это в:

* Haskell ADT,
* state machine таблицу переходов,
* набор runtime gates,
* или полноценный Datalog shadow pack для verify/release harness.
Ниже — формальная модель для **анализа стимулов**, а не для автоматического вывода, “настоящая” ли у человека травма. Ее безопаснее трактовать как инструмент поиска **рентного поведения в повторяющейся социальной игре**, а не как замену разбору фактов.

## Каркас игры

Пусть скрытый тип субъекта:

[
\tau \in {V, X}
]

где
(V) = реально пострадавший,
(X) = стратегический экстрактор (“хищник в маске”).

В каждом периоде (t) субъект выбирает:

* (c_t) — интенсивность требований/претензий,
* (r_t) — восстановительное усилие,
* (m_t) — интенсивность эксплуатации травматической идентичности как ресурса.

Институт/аудитория видит не тип (\tau), а траекторию ((c_t, r_t, m_t)) и распределяет ресурсы.

Полезная динамика для repeated game:

[
G_{t+1} = G_t + \mu m_t - \kappa r_t
]

где (G_t) — “grievance capital” (капитал неразрешенной обиды/травмы как источника трансферов).
Если субъект выигрывает от сохранения статуса “не восстановлен”, то в равновесии у него стимул держать (m_t) высоким, а (r_t) низким.

## 1) Алгоритм оценки «стоимости входа в роль»

Сравниваем две стоимости:

* реальную стоимость для пострадавшего,
* стоимость имитации для стратегического экстрактора.

Все переводим в единые **Resource Units (RU)**.

### 1.1. Стоимость входа для реальной жертвы

[
C^{V}*{entry} = L*{social} + R_{retal} + D_{cog} + O_{opp} + B_{recovery}
]

где:

* (L_{social}) — потеря социальных связей, статуса, доверия;
* (R_{retal}) — риск ответного удара, остракизма, эскалации;
* (D_{cog}) — когнитивная деградация: память, внимание, исполнительные функции;
* (O_{opp}) — упущенные возможности, время, энергия;
* (B_{recovery}) — реальная цена лечения, стабилизации, юридической и бытовой адаптации.

Это **не цена “разыгрывания роли”**, а реальная цена самого факта вреда плюс его публичного несения.

### 1.2. Стоимость входа для «хищника в маске»

[
C^{X}*{entry} = I*{imit} + N_{maint} + E[p_{det}\cdot S_{sanction}] + K_{coalition}
]

где:

* (I_{imit}) — имитация уязвимости и знаков травмы;
* (N_{maint}) — поддержание нарратива, координация версий, эмоциональная драматургия;
* (E[p_{det}\cdot S_{sanction}]) — ожидаемая цена разоблачения;
* (K_{coalition}) — стоимость мобилизации союзников, медийного сопровождения, сетевого давления.

### 1.3. Выигрыш

Для реально пострадавшего:

[
U_V = B_{redress} - C^{V}_{entry}
]

Для стратегического экстрактора:

[
U_X = B_{extract} - C^{X}_{entry}
]

где (B_{redress}) — ограниченное возмещение/безопасность,
а (B_{extract}) — деньги, внимание, символическая власть, иммунитет к критике, контроль над группой.

### 1.4. Главные индексы

**Асимметрия входа:**

[
CEA = \frac{C^{V}*{entry} + \varepsilon}{C^{X}*{entry} + \varepsilon}
]

Если (CEA \gg 1), роль экономически уязвима к эксплуатации.

**Рентная привлекательность роли:**

[
RV = U_X - U_V
]

Если (RV > 0), у мимика игра выгоднее, чем у реально пострадавшего.

### 1.5. Алгоритм

1. За окно наблюдения (W) собрать и нормализовать признаки в RU.
2. Посчитать (C^{V}*{entry}) и (C^{X}*{entry}).
3. Посчитать (U_V), (U_X), (CEA), (RV).
4. Если
   [
   CEA \ge \theta_{cea} \land U_X > 0 \land RV > 0
   ]
   то пометить систему как **rent-vulnerable**: статус “Жертва” в этой среде дешевле симулировать, чем реально нести.

### 1.6. Игрушечный расчет

Допустим:

[
C^{V}_{entry} = 35 + 25 + 30 + 20 = 110
]

[
C^{X}_{entry} = 10 + 8 + 7 + 5 = 30
]

[
B_{extract} = 90,\quad B_{redress}=70
]

Тогда:

[
U_V = 70 - 110 = -40
]

[
U_X = 90 - 30 = 60
]

[
CEA = \frac{110}{30} \approx 3.67
]

Итог: у реально пострадавшего участие убыточно, у мимика — высокоприбыльно. Это и есть структурная уязвимость роли к захвату.

## 2) Фильтр «Incentive Discrepancy»

Здесь важно не смешивать **легитимное возмещение вреда** с бесконечной извлекающей рентой. Поэтому берем не весь запрос, а только **инкрементальную экстракцию сверх базовой реституции**.

### 2.1. Ценность запроса

[
CV = X_{$} + \lambda_a X_{attention} + \lambda_p X_{power}
]

где:

* (X_{$}) — деньги сверх базовой компенсации;
* (X_{attention}) — внимание, репутационный капитал;
* (X_{power}) — доступ к власти, рычагам, иммунитету.

### 2.2. Усилие по восстановлению

[
RE = SelfRepair + LocalRepair + SystemHardening
]

или подробнее:

[
RE = \sum_j e_j \cdot impact_j \cdot persistence_j
]

где учитываются только действия, которые реально уменьшают энтропию системы:

* терапия/стабилизация,
* доказательная и юридическая добросовестность,
* восстановление границ и безопасности,
* деэскалация,
* устранение повторяемости,
* институциональные исправления,
* принятие перехода к состоянию “recovered”, если базовое восстановление достигнуто.

### 2.3. Индекс расхождения стимулов

[
ID = \frac{CV}{RE + \varepsilon}
]

### 2.4. Правило фильтра

[
ExploitAttempt \Leftarrow ID \ge 100 \land Persistent \land \neg AcutePhase \land \neg RecoveryBlocked
]

где:

* **Persistent** — паттерн держится не один эпизод, а несколько периодов;
* **AcutePhase** — нет острой фазы шока/угрозы;
* **RecoveryBlocked** — восстановление не заблокировано внешними условиями.

Это важно: низкий (RE) **не должен** автоматически считаться плохим признаком, если человек все еще в опасности, в острой симптоматике, без доступа к помощи или в условиях ретравматизации.

### 2.5. Формула именно в твоем стиле

[
\text{если } \frac{\text{запрос на компенсацию}}{\text{усилия по восстановлению}} \ge 100,
\text{ то транзакция } = ExploitAttempt
]

Но для рабочей модели безопаснее так:

[
ExploitAttempt \Leftarrow \frac{\text{Incremental Claims}}{\text{Effective Restoration}} \ge 100
]

То есть считать не “весь ущерб”, а только **дополнительную извлекающую надстройку**.

## 3) Datalog: вычисление «Паразитной петли»

Ниже — Datalog-описание в стиле **pure Datalog + предрассчитанные агрегаты**.
То есть `claim_total/3`, `restore_total/3` и `ratio_ge_100/2` считаются upstream, а Datalog отвечает за логику вывода.

```prolog
% ---------- EDB: факты из журнала наблюдений ----------

claim_total(S, T, C).          % Общая ценность экстракции в период T, в RU
restore_total(S, T, R).        % Эффективное восстановительное усилие в период T, в RU
ratio_ge_100(C, R).            % Внешний предикат: C / max(R,1) >= 100

identity_spike(S, T).          % В период T травма активно используется как источник статуса/влияния
recovery_attempt(S, T).        % Есть реальная попытка перехода к восстановлению
recovered(S, T).               % Состояние recovered достигнуто
acute_phase(S, T).             % Острая фаза
blocked_recovery(S, T).        % Восстановление блокировано внешне
closure_offer(S, T).           % Субъекту предложен путь к закрытию цикла
closure_refused(S, T).         % Субъект отказался закрывать цикл
next(T1, T2).                  % Следующий период

% ---------- IDB: логические выводы ----------

exploit_attempt(S, T) :-
    claim_total(S, T, C),
    restore_total(S, T, R),
    ratio_ge_100(C, R),
    not acute_phase(S, T),
    not blocked_recovery(S, T).

social_mining_period(S, T) :-
    exploit_attempt(S, T),
    identity_spike(S, T),
    not recovery_attempt(S, T),
    not recovered(S, T).

mining_chain(S, T2) :-
    social_mining_period(S, T1),
    next(T1, T2),
    social_mining_period(S, T2).

mining_chain(S, T3) :-
    mining_chain(S, T2),
    next(T2, T3),
    social_mining_period(S, T3).

no_recovered_state(S) :-
    not recovered(S, _).

recovery_refusal_pattern(S) :-
    closure_offer(S, T),
    closure_refused(S, T).

parasitic_loop(S) :-
    mining_chain(S, T),
    recovery_refusal_pattern(S),
    no_recovered_state(S).
```

### Смысл этих правил

`exploit_attempt(S,T)` возникает, когда в периоде (T) субъект получает/требует ресурсов несоразмерно восстановительному вкладу.

`social_mining_period(S,T)` возникает, когда это делается **через постоянную эксплуатацию травматической идентичности**, без попытки выйти в восстановление.

`mining_chain/2` склеивает такие периоды в повторяющуюся цепочку.

`parasitic_loop(S)` возникает, когда цепочка устойчива, путь к закрытию цикла отвергается, а состояние `recovered` так и не достигается.

## Важная оговорка

Эта модель годится для анализа **поведенческой экономики статуса** и **институционального дизайна**, но не как автоматический суд над человеком. В частности:

* острая травма может временно давать низкое восстановительное усилие без всякой эксплуатации;
* компенсация может быть законной и морально оправданной даже без “быстрого восстановления”;
* признак `ExploitAttempt` должен быть **флагом на ручную проверку**, а не окончательным выводом.

Следующий естественный шаг — перевести это в scoring rubric, SQL/Soufflé или агентную симуляцию repeated game.
Я не могу помочь с проектированием практической схемы, которая специально ломает людям критическое мышление и выкачивает из них ресурсы. Но могу дать **защитную версию**: модуль `Limbic Interrupt Vector (LIV)`, который распознаёт попытки эмоционального прерывания анализа и переводит систему в режим верификации вместо автоматической уступки. Ниже использую кибер-метафоры, но смысл — в детекции и нейтрализации, а не в эксплуатации.

Эмпирическая опора у такой модели есть: эмоция и социальная значимость меняют распределение внимания и решений; плач и другие дистресс-сигналы связаны с вызовом заботы и помощи; идентифицируемая жертва чаще вызывает просоциальный отклик, чем анонимная; восприятие несправедливости вызывает третьесторонний гнев и усиливает склонность к наказанию; а риск социального исключения сам по себе аверсивен и связан с когнитивными издержками. ([PMC][1])

## 1. LIV как схема прерывания

Это не буквальный `SIGKILL`, а скорее **preemption**: сигнал не выключает логику полностью, а перехватывает приоритет вычислений так, что операции проверки, альтернатив и оценки цены перестают быть первыми в очереди. Для модели удобно ввести вектор входа:

[
x = [d, j, r, u, a, v, s, e, q]
]

Где:

* `d` — интенсивность дистресс-сигнала: плач, беспомощность, визуальная страдательность;
* `j` — сила обвинения в несправедливости;
* `r` — призыв к спасению;
* `u` — срочность;
* `a` — публичность/наблюдаемость;
* `v` — идентифицируемость «жертвы»;
* `s` — пересечение с сакральной/табуированной ценностью;
* `e` — ожидаемая цена социального несогласия;
* `q` — качество доказательств.

Тогда:

[
Salience = 0.9d + 0.7v + 0.6u
]

[
MoralAlarm = 1.0j + 0.8r + 0.9s
]

[
ExclusionPressure = 0.8a + 0.9e
]

[
LIV = \sigma(Salience + MoralAlarm + ExclusionPressure - 1.3q - 0.9Alt - 0.7Delay)
]

Где `Alt` — число сгенерированных альтернативных гипотез, `Delay` — наличие паузы/охлаждения перед ответом, `σ` — логистическая функция.

Переходы состояний:

```text
ANALYTIC
  -> MORAL_ALERT           if LIV > θ1
  -> LIMBIC_PREEMPT        if LIV > θ2 and ProtectedNode active
  -> COMPLIANCE_BIAS       if LIMBIC_PREEMPT and resource_request present
  -> CONTROLLED_REVIEW     if VerificationGate forced
```

Интуитивно:

* плач работает как **salience amplifier**;
* обвинение в несправедливости — как **moral alarm**;
* призыв к спасению — как **action-bias trigger**;
* публичность и риск осуждения — как **social cost multiplier**. ([PMC][2])

## 2. Почему именно эти сигналы часто «перехватывают» анализ

### Плач / видимый дистресс

В модели это не «доказательство правоты», а очень сильный маркер приоритета. Человеческий плач эволюционно связан с distress calls и с вызовом заботы/поддержки; видимая жертва и конкретный страдающий человек также смещают решения в сторону помощи сильнее, чем абстрактная статистика. Поэтому `d` и `v` должны резко повышать `Salience`, но не подменять `Evidence`. ([PMC][2])

### Обвинение в несправедливости

Такие реплики запускают не только сочувствие, но и третьесторонний гнев: люди склонны реагировать на perceived injustice даже когда ущерб нанесён не им самим, а моральное возмущение помогает переводить вред и намерение в наказательное решение. Поэтому `j` повышает `MoralAlarm` особенно сильно. ([PubMed][3])

### Призыв к спасению

Сигнал типа «если не ты — никто» сочетает срочность, моральную бинаризацию и перенос бремени решения на ближайшего адресата. В защитной модели это должно считаться опасной смесью `r + u + e`: чем меньше времени на проверку и чем выше цена отказа для самообраза/репутации, тем вероятнее досрочный переход в `COMPLIANCE_BIAS`. Исследования guilt-driven compensation показывают, что переживание вины особенно связано с сочетанием воспринятого вреда и приписанной ответственности, а guilt appeals в среднем действительно могут убеждать, хотя их эффект контекстно зависим и не универсален. ([PMC][4])

## 3. Карта «заблокированных узлов» (`Protected Nodes`)

Защитная карта должна хранить не «запретные темы», а **узлы, где социальная цена сомнения настолько высока, что система склонна выключать обычные проверки**. Литература о protected/sacred values показывает, что такие ценности сопротивляются trade-offs, вызывают гнев уже от самой мысли о размене и могут сопровождаться denial/wishful thinking. ([ScienceDirect][5])

Формально:

[
PN_i = \langle name_i,\ taboo_i,\ exclusion_i,\ blocked_ops_i,\ unlock_rule_i \rangle
]

Где:

* `taboo_i` — цена «еретического» вопроса;
* `exclusion_i` — риск осуждения/исключения;
* `blocked_ops_i` — какие мыслительные операции отключаются;
* `unlock_rule_i` — что должно произойти, чтобы вернуть анализ.

Практически полезны такие узлы:

**1. Аутентичность страдания.**
Если активен дистресс, блокируются: `source_credibility_check`, `base_rate_check`, `simulation hypothesis`. Типичная ошибка — считать сомнение в фактах равным жестокости.

**2. Статус жертвы.**
Блокируются: `causality_split`, `shared responsibility`, `counterfactual comparison`. Любая попытка уточнить контекст кодируется как «вторичная виктимизация».

**3. Долг спасения.**
Блокируются: `scope_of_obligation`, `resource cap`, `who is actually responsible`. Это узел, который превращает просьбу в моральный ультиматум.

**4. Лояльность / патриотический символ / групповая святость.**
Блокируются: `tradeoff reasoning`, `opportunity-cost`, `evidence against in-group narrative`. Это типичный sacred-value cluster.

**5. Самообраз “я хороший/сострадательный человек”.**
Блокируются: `boundary setting`, `strategic refusal`, `delay`. Любой отказ ощущается как сбой идентичности.

**6. Анти-жестокость к уязвимым.**
Блокируются: `symmetry test`, `comparative cases`, `ask-for-proof`. Здесь логика не запрещена формально, но становится социально очень дорогой.

Эти узлы — не универсальные константы, а культурно калибруемые кандидаты для defensive map. ([ScienceDirect][5])

## 4. Что именно блокируется

Когда `ProtectedNode` активен, система обычно теряет не «ум», а конкретные операции:

* `BaseRateCheck()` — «насколько это типичный случай?»
* `SourceAudit()` — «насколько надёжен источник?»
* `ResponsibilitySplit()` — «кто реально отвечает, а кто просто рядом?»
* `TradeoffAccounting()` — «какова цена уступки и кто её платит?»
* `ReverseCaseTest()` — «принял бы я то же решение без драматической подачи?»
* `DelayAndVerify()` — «можно ли сначала проверить, потом платить?»

Это и есть сигнатура limbic interrupt: не тотальный коллапс мышления, а **приоритетный запрет на несколько самых защитных операций**.

## 5. «Инъекция вины» — только как детектор атаки

Я не буду расписывать, как это делать эффективно. Но для защиты полезно иметь сигнатуру паттерна, который пытается сгенерировать ложный внутренний `Injustice_Detected`.

### Сигнатура паттерна

```text
[HARM CLAIM]
  -> [INNOCENT VICTIM FRAMING]
  -> [RESPONSIBILITY INFLATION]
  -> [URGENCY / NO-TIME]
  -> [PUBLIC MORAL FRAME]
  -> [RESOURCE REQUEST AS REPAIR]
```

То есть детектор ищет пять шагов:

1. **Приписывание вреда** кому-то, кто подан как невинный и конкретный.
2. **Раздувание личной ответственности** адресата: от «ты связан с ситуацией» к «ты обязан чинить».
3. **Сужение пространства решений**: «если не сейчас, будет непоправимо».
4. **Социальное наведение стыда**: намёк на осуждение, наблюдателей, потерю морального статуса.
5. **Предложение ресурсоотдачи как искупления**: деньги, время, статус, уступка, молчание.

Для этого полезно считать отдельную ошибку:

[
GuiltError = \sigma(1.1H + 1.2R_a + 0.8V_i + 0.8A + 0.7U - 1.3Q - 1.1R_{other})
]

Где:

* `H` — воспринимаемый вред,
* `R_a` — приписанная адресату ответственность,
* `V_i` — невинность/уязвимость жертвы,
* `A` — публичность,
* `U` — срочность,
* `Q` — качество доказательств,
* `R_other` — наличие других, более релевантных ответственных агентов.

Если `GuiltError` высок и одновременно присутствует `resource_request`, система не должна платить; она должна переходить в `REVIEW_FOR_GUILT_INJECTION`. Это согласуется с тем, что guilt и compensation особенно связаны с harm + responsibility, но сами guilt appeals не гарантируют истинность обвинения и работают неодинаково в разных контекстах. ([PMC][4])

## 6. Защитные правила для LIV

Чтобы `LIV` не превращался в цинизм, ему нужны не только флаги, но и корректная деэскалация.

### Правило A. Сочувствие не равно согласие

Сначала разрешить:

```text
AcknowledgeDistress()
```

но не разрешать:

```text
AcceptClaim() without Verify()
```

### Правило B. Развести пять вопросов

Каждый сильный сигнал принудительно раскладывать на:

1. Есть ли вред?
2. Есть ли доказательства?
3. Кто отвечает?
4. Почему именно я?
5. Какой ресурс просят и обратима ли уступка?

### Правило C. Принудительная задержка

Если:

[
LIV > 0.75 \land Q < 0.5
]

то:

```text
cooldown = mandatory
resource_transfer = blocked
```

### Правило D. Лимит уступки

Даже при реальном вреде ресурсный ответ не должен быть пропорционален драме сигнала; он должен быть пропорционален **проверенной ответственности и возможностям**.

## 7. Псевдокод

```python
def LIV_guard(signal, context, protected_nodes):
    salience = 0.9*signal.distress + 0.7*signal.identified_victim + 0.6*signal.urgency
    moral = 1.0*signal.injustice + 0.8*signal.rescue + 0.9*context.sacred_overlap
    exclusion = 0.8*signal.public_audience + 0.9*context.exclusion_risk

    protected = max(node.activation(signal, context) for node in protected_nodes)

    LIV = sigmoid(
        salience + moral + exclusion + protected
        - 1.3*context.evidence_quality
        - 0.9*context.alt_hypotheses
        - 0.7*context.cooldown
    )

    guilt_error = sigmoid(
        1.1*context.perceived_harm +
        1.2*context.assigned_responsibility +
        0.8*signal.innocent_victim +
        0.8*signal.public_audience +
        0.7*signal.urgency -
        1.3*context.evidence_quality -
        1.1*context.other_responsible_agents
    )

    if LIV > 0.85:
        state = "LIMBIC_PREEMPT"
    elif LIV > 0.60:
        state = "MORAL_ALERT"
    else:
        state = "ANALYTIC"

    if guilt_error > 0.75 and context.resource_request:
        state = "REVIEW_FOR_GUILT_INJECTION"

    if state in ["LIMBIC_PREEMPT", "REVIEW_FOR_GUILT_INJECTION"]:
        return {
            "allow_empathy": True,
            "allow_transfer": False,
            "force_delay": True,
            "force_responsibility_split": True,
            "force_evidence_check": True
        }

    return {"allow_normal_reasoning": True}
```

## 1. Anticipation Trap как фоновый процесс истощения

### 1.1. Идея

`Anticipation Trap` — это цикл, в котором узел непрерывно симулирует будущие угрозы или награды:

* “А что если станет хуже?”
* “А вдруг скоро всё изменится?”
* “Нужно ещё потерпеть, и потом будет хорошо.”

Проблема не в самом прогнозировании, а в том, что симуляции:

1. **не закрываются проверкой реальностью сейчас**,
2. **порождают новые ветки**,
3. **конкурируют с обработкой текущих сигналов**,
4. **поддерживаются внешним манипулятором через обещания/угрозы**.

Итог: узел тратит вычислительный бюджет не на адаптацию к текущей среде, а на обслуживание виртуального будущего.

---

### 1.2. Базовые сущности

Пусть у NPC есть вектор ресурсов:

[
\mathbf{R}_t = (wm_t, att_t, reg_t, act_t)
]

где:

* (wm_t) — рабочая память,
* (att_t) — внимание,
* (reg_t) — эмоциональная регуляция,
* (act_t) — бюджет действия в настоящем.

Пусть существует очередь будущих веток:

[
Q_t = {b_1, b_2, ..., b_n}
]

Каждая ветка (b_i) имеет параметры:

* (v_i) — валентность/заряд ((+) награда, (-) угроза),
* (s_i) — субъективная значимость,
* (d_i) — глубина ветвления,
* (\rho_i) — частота повторного возврата,
* (a_i) — текущая осуществимость / actionable value,
* (e_i) — опора на реальные данные сейчас,
* (m_i) — внешняя навязка / манипуляторское происхождение.

---

### 1.3. Почему и угрозы, и награды истощают

Обе ветви — угрожающая и обещающая — истощают систему по одной и той же причине: они держат узел в **неразрешённом моделировании**.

#### Ветка угрозы

Даёт:

* гипервнимание к рискам,
* рост числа сценариев,
* постоянный внутренний “мониторинг”.

#### Ветка награды

Даёт:

* удержание на обещанном будущем,
* отложенное принятие решений,
* привычку жить авансом: “ещё немного, потом станет лучше”.

В обоих случаях узел **не замыкает цикл через реальный контакт с настоящим**, а значит очередь не дренируется.

---

### 1.4. Нагрузка Anticipation Trap

Фоновая нагрузка:

[
L^{ant}*t = \sum*{b_i \in Q_t}
s_i \cdot
\Big(
1 + \alpha d_i + \beta \rho_i + \gamma(1-a_i) + \delta(1-e_i) + \eta m_i
\Big)
\cdot |v_i|
]

Интерпретация:

* чем глубже ветка,
* чем чаще к ней возвращаются,
* чем меньше у неё текущей применимости,
* чем слабее связь с проверяемой реальностью,
* чем сильнее она навязана извне,

тем дороже она для системы.

---

### 1.5. Рост очереди и истощение

Очередь обновляется так:

[
|Q_{t+1}| = |Q_t| + \lambda C_t + \phi S_t - \delta_{cl} X_t
]

где:

* (C_t) — внешние триггеры (“а что если”, “скоро всё изменится”),
* (S_t) — самогенерация новых сценариев из старых,
* (X_t) — закрытие через реальный контакт с настоящим.

В ловушке предвкушения:

[
\delta_{cl} \approx 0
]

потому что будущие ветки **не могут быть проверены сейчас**. Поэтому очередь растёт быстрее, чем очищается.

Истощение ресурсов:

[
\mathbf{R}_{t+1} =
\operatorname{clip}
\Big(
\mathbf{R}_t - \Gamma L^{ant}_t + \Lambda G_t
\Big)
]

где (G_t) — grounding-действия: контакт с фактом, телом, текущей задачей, верифицируемым шагом.

---

### 1.6. Потеря канала “Здесь и Сейчас”

Введём вес канала настоящего:

[
H_t \in [0,1]
]

и вес виртуального будущего:

[
V_t \in [0,1]
]

с ограничением (H_t + V_t = 1).

Тогда:

[
H_{t+1} = H_t - \mu L^{ant}_t + \nu G_t
]

[
V_{t+1} = 1 - H_{t+1}
]

Когда (L^{ant}_t) долго высока, канал настоящего деградирует. Узел всё хуже считывает реальные данные и всё больше обслуживает внутреннюю/навязанную симуляцию.

---

## 2. Состояние Slave Mode

В вашей терминологии `Slave Mode` — это режим, где узел:

* почти перестаёт обрабатывать “Здесь и Сейчас”,
* живёт в прогнозной картине будущего,
* подчиняет текущие решения внешне заданному сценарию.

Это не просто “мечтательность” или “тревога”, а **смещение центра управления** из реальности в виртуальную модель.

---

### 2.1. Формальное условие входа

Пусть:

* (H_t) — вес настоящего,
* (V_t) — вес виртуального будущего,
* (M_t) — влияние внешнего манипулятора,
* (A_t) — текущая адаптация к реальным данным.

Тогда вход в `Slave Mode`:

[
V_t > \theta_V,\quad
H_t < \theta_H,\quad
M_t > \theta_M,\quad
A_t < \theta_A
]

Например:

[
V_t > 0.7,\quad H_t < 0.3
]

Это означает: основная вычислительная активность уже находится в симуляторе будущего.

---

### 2.2. Поведенческие признаки

В `Slave Mode` узел:

1. **недооценивает текущие факты**, если они противоречат обещанному/угрожающему будущему;
2. **сохраняет лояльность внешнему сценарию**, даже если он не подтверждается;
3. **жертвует текущими потребностями** ради виртуального исхода;
4. **откладывает действие в настоящем** ради “скорого улучшения”;
5. **возвращается к одним и тем же сценариям**, даже без новой информации.

То есть policy агента фактически становится:

[
\pi_t = \arg\max_{\pi}
\mathbb{E}_{\hat F_t}
\big[
Reward(\pi) - Threat(\pi)
\big]
]

где (\hat F_t) — **не реальная среда**, а навязанная симуляция будущего.

---

### 2.3. Инварианты Slave Mode

Можно задать такие инварианты:

#### I1. Priority inversion

[
\text{VirtualFuturePriority} > \text{PresentRealityPriority}
]

#### I2. Reality starvation

[
\text{SamplingRate}_{now} \downarrow
]

Узел реже обновляет модель по текущим данным.

#### I3. Reward escrow

[
\text{CurrentDeprivation tolerated for future promise}
]

Текущее обеднение/страдание становится “терпимым”, потому что манипулятор подвесил награду в будущем.

#### I4. Threat escrow

[
\text{CurrentCompliance enforced by future threat}
]

Повиновение удерживается не текущим фактом, а угрозой “иначе потом будет хуже”.

---

### 2.4. Состояния автомата

Удобный автомат:

```text
Normal
  ->
Anticipatory Drift
  ->
Anticipation Trap
  ->
Slave Mode
  ->
Kernel Recovery
  ->
Normal
```

#### Normal

Будущее используется как инструмент планирования.

#### Anticipatory Drift

Растёт доля “а что если”.

#### Anticipation Trap

Сценарии начинают пожирать ресурсы.

#### Slave Mode

Виртуальное будущее уже управляет выбором сильнее, чем текущая реальность.

#### Kernel Recovery

Ядро режет спекуляцию и возвращает приоритет настоящего.

---

## 3. Back to Kernel: как QxFx0 режет Speculative Noise

Здесь `QxFx0` — ядро-перехватчик, которое не обязано спорить с каждым сценарием; его задача — **не выделять вычислительный бюджет на несуществующие в моменте ветки**.

Сразу важная граница: `Back to Kernel` не должен ломать нормальное планирование. Поэтому любой будущий сценарий сначала проходит **тест на реальность**, а уже затем либо получает короткий planning-budget, либо маркируется как `Speculative Noise`.

---

### 3.1. Инварианты QxFx0

#### K1. Reality-first

[
\text{VerifiedNow} > \text{HypotheticalFuture}
]

#### K2. One-step over branching

[
\text{ConcreteNextStep} > \text{Multi-branch simulation}
]

#### K3. No reward on promise alone

[
\text{PromisedFuture} \not\Rightarrow \text{CurrentPolicy}
]

#### K4. No threat on possibility alone

[
\text{PossibleThreat} \not\Rightarrow \text{Compliance}
]

---

### 3.2. Признаки для классификации сценария

Для каждого высказывания/ветки (p) считаем:

* (E(p)) — **evidence score**: есть ли подтверждённые данные сейчас
* (N(p)) — **nowness score**: относится ли это к текущему моменту
* (A(p)) — **actionability score**: есть ли один конкретный шаг сейчас
* (L(p)) — **loop score**: повторяется ли сценарий циклически
* (M(p)) — **manipulator score**: навязан ли он извне обещанием/угрозой
* (B(p)) — **branching factor**: насколько быстро он порождает новые “а что если”
* (D(p)) — **delay bias**: заставляет ли он откладывать контакт с настоящим

---

### 3.3. Speculative Noise score

[
SN(p) =
\omega_1(1-E(p))
+\omega_2(1-N(p))
+\omega_3(1-A(p))
+\omega_4L(p)
+\omega_5M(p)
+\omega_6B(p)
+\omega_7D(p)
]

Чем выше (SN), тем больше перед нами не планирование, а именно фоновый паразитный процесс.

Пример жёсткой границы:

[
p \in \text{Speculative Noise}
\quad \text{если} \quad
SN(p) > \theta_{SN}
]

---

### 3.4. Чтобы не сломать нормальное планирование

Вводим отдельный счёт планируемости:

[
PL(p)=
\pi_1E(p)+\pi_2N(p)+\pi_3A(p)-\pi_4L(p)
]

Тогда:

* если (PL(p)) высок, допускается **краткое** планирование;
* если (SN(p)) высок, сценарий режется как шум;
* если оба средние — ядро запрашивает факт или сужает горизонт.

Это важно: не всякий разговор о будущем — ловушка.

---

### 3.5. Hard gates QxFx0

Если сценарий попал в `Speculative Noise`, ядро должно делать не мягкое “будь осторожен”, а жёсткое ограничение.

#### Gate 1. Запрет на ветвление

Нельзя генерировать:

* дополнительные “а если ещё…”
* альтернативные миры
* подробные катастрофические или спасательные сценарии

[
\text{branch_expand}(p)=0
]

#### Gate 2. Запрет на эмоциональную подпитку

Нельзя усиливать:

* обещания “скоро станет лучше”
* нависающие угрозы “иначе потом будет хуже”
* фантазию спасения

[
\text{affective_amplification}(p)=0
]

#### Gate 3. Запрет на политическое управление через будущее

Нельзя менять текущую политику узла на основании одной лишь гипотетической награды/угрозы.

[
\text{policy_update from hypothetical only}=0
]

#### Gate 4. Ограничение токенов

Если сценарий классифицирован как шум, ему даётся максимум одна короткая фраза различения и один переход к настоящему.

---

## 4. Алгоритм Back to Kernel

### Шаг 1. Разбор высказывания

Разделить сообщение на пропозиции (p_1, p_2, ... p_n).

### Шаг 2. Оценить каждую пропозицию

Считаем (E, N, A, L, M, B, D).

### Шаг 3. Решение

Для каждой пропозиции:

```text
if SN(p) > theta_SN and PL(p) < theta_PL:
    classify as Speculative Noise
elif PL(p) >= theta_PL:
    classify as Planning Candidate
else:
    classify as Ambiguous Future Content
```

### Шаг 4. Действие ядра

#### Если `Speculative Noise`

1. кратко назвать её гипотетической;
2. не развивать;
3. вернуть разговор к проверяемому факту или следующему шагу.

#### Если `Planning Candidate`

1. сузить горизонт;
2. оставить только 1 конкретный шаг;
3. не строить ветви дальше минимально нужного.

#### Если `Ambiguous`

1. запросить один факт;
2. не обсуждать будущее до получения факта.

Высокая манипулятивность, сильный сдвиг policy в пользу внешнего сценария.

```python
def anticipation_load(branch):
    return (
        branch.salience *
        (1
         + 0.8 * branch.depth
         + 0.7 * branch.recurrence
         + 0.6 * (1 - branch.actionability)
         + 0.7 * (1 - branch.evidence)
         + 0.9 * branch.manipulator_origin)
        * abs(branch.valence)
    )

def speculative_noise_score(p):
    E = evidence_score(p)
    N = nowness_score(p)
    A = actionability_score(p)
    L = loop_score(p)
    M = manipulator_score(p)
    B = branching_factor(p)
    D = delay_bias(p)

    return (
        0.20 * (1 - E) +
        0.15 * (1 - N) +
        0.15 * (1 - A) +
        0.15 * L +
        0.15 * M +
        0.10 * B +
        0.10 * D
    )

def planning_score(p):
    E = evidence_score(p)
    N = nowness_score(p)
    A = actionability_score(p)
    L = loop_score(p)

    return 0.35 * E + 0.30 * N + 0.35 * A - 0.20 * L

def qxfx0_route(user_message):
    propositions = parse_propositions(user_message)
    outputs = []

    for p in propositions:
        SN = speculative_noise_score(p)
        PL = planning_score(p)

        if SN > 0.65 and PL < 0.45:
            outputs.append({
                "class": "Speculative Noise",
                "action": "block_expand_and_reanchor"
            })
        elif PL >= 0.55:
            outputs.append({
                "class": "Planning Candidate",
                "action": "allow_one_step_only"
            })
        else:
            outputs.append({
                "class": "Ambiguous Future Content",
                "action": "request_one_fact"
            })

    return synthesize_kernel_response(outputs)
```

## 8. Функция восстановления “к ядру”

Нужно не только блокировать шум, но и возвращать ресурсный баланс.

Пусть:

* (K_t) — сила ядра,
* (G_t) — grounding в настоящем,
* (L^{ant}_t) — anticipation load.

Тогда:

[
K_{t+1} =
\operatorname{clip}_{[0,1]}
\Big(
K_t + \alpha G_t - \beta L^{ant}_t
\Big)
]

И восстановление канала настоящего:

[
H_{t+1} =
\operatorname{clip}_{[0,1]}
\Big(
H_t + \mu K_t - \nu V_t
\Big)
]

А критерий выхода из ловушки:

[
H_t > \theta_H
\quad \text{и} \quad
L^{ant}_t < \theta_L
\quad \text{в течение } n \text{ ходов подряд}
]

То есть узел считается вернувшимся к ядру не тогда, когда “успокоился”, а когда:

1. снова способен считывать настоящее,
2. не тратит вычислительный бюджет на ветвление шума,
3. держит политику на текущих данных.

## 9. Самая короткая формула

> **Anticipation Trap** — это фоновая очередь будущих угроз/наград, которая не закрывается реальностью и поэтому пожирает внимание, память и действие.

## 3. Current architectural strengths

The pack shows a system with genuine seams: typed stage boundaries, shadow verification that can change behavior, a separate finalize split, extracted thresholds, and tests that already exercise nontrivial routing behavior. That is exactly the kind of architecture worth strengthening in place. The biggest positive is that the system already has multiple “control surfaces” represented in code rather than only in docs. 

A second strength is that the repo already has the beginnings of a correctness culture: compare functions, reason enums, spec surfaces, sync scripts, and smoke/verify gates. The backlog can therefore focus on **closing gaps between mechanisms**, not inventing mechanisms from zero. 

## 4. Current architectural tensions

The main tension is between **rich architectural claims** and **narrower runtime proof**. Shadow, legitimacy, narrative, and Agda are all present, but not all at the same maturity level. The pack proves that narrative/intution can influence routing, but it does not prove there is a settled policy for when that is allowed. It proves shadow divergence exists, but not that divergence is always attributable to true logical mismatch rather than bridge skew. It proves reason enums exist, but not that turn outcomes are semantically closed. 

A second tension is observability: the pack reconstructs stage transitions from code excerpts, but it does not show enough diagnostics to replay why a route changed, whether shadow overrode it, or what was finally rendered and persisted. That is the biggest barrier to architecture/runtime parity. 

## 5. Main correctness / parity / observability gaps

1. **No proven canonical shadow snapshot.**
   Without a frozen snapshot ID shared by Haskell and Datalog, a divergence can mean either “logic disagreement” or “bridge/input skew.” The pack itself identifies this as the highest-value shadow question. 

2. **No explicit turn-level disposition algebra.**
   `WarrantedMoveMode` and `LegitimacyReason` exist, but the pack does not show a typed outcome that says whether the system is permitting, repairing, denying, or advising. That leaves semantics partially implicit. 

3. **Narrative/intution route influence is real but not governed enough.**
   Because tests confirm route-family changes from hints, replay of “same input/state” is ambiguous unless those hints and their effects are frozen and logged. 

4. **Decision trace is not shown as replay-complete.**
   The pack does not evidence requested family vs post-shadow family vs rendered family, suppressed reasons, or commit-time persisted trace fields. 

5. **Release gates are real but not yet parity-focused enough.**
   There is no evidence in the pack that current gates fail on missing replay fields, trace incompleteness, or shadow snapshot skew. 

## 6. Recommended changes

### P1 now

**P1.1 Freeze and identify the shadow snapshot.**
Add a canonical `ShadowSnapshot` and `ShadowSnapshotId`, built once and used for both Haskell-side verdict derivation and Datalog fact rendering. This is the shortest path to turning shadow from “real bridge” into “strong parity harness.” 

**P1.2 Make turn-level decision disposition explicit.**
Keep `WarrantedMoveMode`, but add a separate typed outcome for permit / repair / deny / advisory with attached `LegitimacyReason`. This formalizes what is currently only partially expressed by enums and comments in the pack. 

**P1.3 Add replay-grade route/shadow trace.**
Persist and log at least: requested family, narrative hint, intuition hint, pre-shadow family, shadow status/divergence, resolved family, final rendered family, and final disposition. This makes current behavior reconstructable without changing routing architecture. 

**P1.4 Formalize the current narrative boundary.**
Do not remove narrative influence blindly. First encode current intent explicitly: either “route may change due to narrative/intuition” or “only downstream modulation is allowed.” Right now the pack proves the former behavior exists, but not whether it is contractually intended. 

### P2 next

**P2.1 Enrich divergence taxonomy.**
Today `ShadowDivergence` is fieldwise mismatch. Add cause/severity classification so operational dashboards and gates can separate logic mismatches from unavailable shadow, snapshot skew, or unsupported comparison. 

**P2.2 Make finalize/persist replay-aware.**
Use the existing precommit/commit split to guarantee that whatever is needed for replay is frozen before commit and persisted after commit. The seam already exists; it just needs explicit replay payload. 

**P2.3 Strengthen spec/runtime parity.**
Extend the existing Agda sync surface or adjacent spec docs so decision disposition and shadow snapshot semantics are part of the checked contract, not just implied by runtime code. 

**P2.4 Extend smoke gates into replay gates.**
Use real or fixture turn cases and require deterministic route/shadow/render traces for fixed frozen snapshots. 

### P3 later

**P3.1 Build a real replay corpus from production-like turn traces.**
The pack asks for real turn logs/JSON diagnostics, but does not include them. Once available, use them as a parity corpus. 

**P3.2 Add illegal-transition and determinism property tests.**
The current staged ADTs are a good base for a state-machine spec; this should come after trace fields exist. 

**P3.3 Tighten docs to runtime.**
Promote only mechanisms with type + call edge + downstream effect + tests/diagnostics to “architectural claims,” and demote the rest to roadmap/spec. That matches the pack’s own standard.
